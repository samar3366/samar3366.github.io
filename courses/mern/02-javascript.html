① Variables — boxes that hold values Basics

A variable is a labelled box you store a value in so you can use it later. Imagine a row of lockers: each locker has a name written on the front (the variable name) and something inside it (the value). When you need that something again, you don't have to remember where it physically is — you just say its name and JavaScript fetches it for you.

In modern JavaScript you create variables with two keywords: let (a box whose contents you can swap out later) and const (a box that's sealed once you put something in — you can't reassign it). There's an older keyword, var, that you'll see in old tutorials; we'll explain later why we avoid it.

Why it matters: almost every line of real code reads from or writes to a variable. Getting comfortable here makes everything else click. Why const by default? Most values in a program never need to change, and using const tells everyone reading your code "this won't move", which prevents a whole category of accidental bugs. Reach for let only when you genuinely need to reassign the value (a running total, a counter, a loop index).

⚠️

Naming rules: no spaces, start with a letter (or _/$), and use camelCase for multi-word names (firstName, totalPrice). Names are case-sensitive: score ≠ Score.

(Note: with objects and arrays, const stops you reassigning the box, but you can still change what's inside — more on that in the Immutability section.)

② Data types — the kinds of values Basics

Every value in JavaScript has a type — a category that tells the language what kind of thing it is and what you can do with it. You can add two numbers, but "adding" two booleans makes no sense. JavaScript figures out the type automatically from how you write the value (you never declare it), which is why it's called a dynamically typed language. These five types cover almost everything you'll do:

TypeExampleUsed for
String"hello"Text (always in quotes)
Number42, 3.14Any number (no quotes)
Booleantrue / falseYes/no, on/off decisions
Array[1, 2, 3]An ordered list of values
Object{ name: "Sam" }A labelled bundle of values

Two more special values round things out: null (an intentional "nothing here" that you set on purpose) and undefined (JavaScript's way of saying "this was never given a value"). You'll meet them constantly — for example, a variable you declared but didn't assign is undefined, and a missing object property is undefined too.

💡

Arrays and objects are technically both "objects" under the hood — so typeof [1,2,3] surprisingly returns "object", not "array". That's a famous quirk. To check specifically for an array, use Array.isArray(value) instead of typeof.

③ Operators — doing things with values Basics

An operator is a symbol that combines values to produce a new value. You already know most of them from maths class (+, -, *, /). JavaScript adds a few that are worth meeting early: % (the "remainder" or modulo operator) and the comparison and logic operators that power every decision your program makes.

The % operator gives you the remainder after dividing. 10 % 3 is 1 because 3 goes into 10 three times with 1 left over. It looks niche but it's incredibly handy: n % 2 === 0 tells you whether n is even, and remainder is how you wrap values around (like clock hours or alternating row colours).

The logic operators are "short-circuit": && stops at the first falsy value, and || stops at the first truthy one. ⑤ Making decisions — if / else, ternary, switch Basics

So far our code runs top to bottom, doing the same thing every time. Real programs need to react: show an error if a password is too short, give a discount if the cart is over ₹1000, greet a user by name only if they're logged in. That branching — choosing a path based on a condition — is what if/else is for. This is where programs start to feel "smart".

The shape is always the same: if (condition) { ... }. The condition is anything that evaluates to truthy or falsy (which you just learned!). If it's truthy, the block in { } runs; otherwise it's skipped. Add else if to test more conditions in order, and a final else as the catch-all "none of the above".

⑥ Functions — reusable blocks of code Basics Intermediate

A function is a named recipe — a chunk of code you write once and reuse as many times as you like. You define it once, then call it (run it) whenever you need it, possibly with different inputs each time. This is the single most important idea for keeping code organised: instead of copying and pasting the same ten lines in five places, you wrap them in a function and call it five times. Fix a bug once, and it's fixed everywhere.

Functions can take inputs (called parameters when you define them, arguments when you pass them in) and hand back an output using the return keyword. The moment a return runs, the function stops and gives that value back to whoever called it. A function with no return hands back undefined.

Why it matters: React components are functions. Node route handlers are functions. Every .map() and .filter() takes a function. If you're fluent with functions, the rest of this course is mostly learning where to put them.

⑦ Scope & hoisting — where variables live Intermediate

Scope answers the question: "from where in my code can I see this variable?" Think of it like rooms in a house. Something on the kitchen table (a global variable) can be seen from anywhere. Something inside a closed drawer (a variable inside a function) can only be seen from inside that drawer. Trying to read it from another room gives you an error — the variable simply doesn't exist out there.

There are three levels of scope to know:

ScopeWhere it's visibleCreated by
GlobalEverywhere in the fileDeclared at the top level
FunctionOnly inside that functionDeclared inside a function
BlockOnly inside the nearest { }let/const inside any { }

The big practical point: let and const are block-scoped, meaning they only exist inside the nearest pair of curly braces { } — an if block, a for loop body, any block. The old var keyword is function-scoped, which leaks variables out of blocks and causes surprising bugs. That's the main reason we never use var today.

Console
⚠️

The classic var loop bug: with var i in a loop that schedules callbacks, every callback ends up sharing the same i and prints the final value. With let i, each loop turn gets its own fresh i and it works as expected. One more reason to always use let/const.

Hoisting — a gentle heads-up

JavaScript does a quick "scan" of your code before running it, and during that scan it notices your function and variable declarations. The effect is that function declarations (function foo() {}) can be called even on a line above where they're written — they're "hoisted" to the top. Handy, but not something to rely on. Variables declared with let/const are not usable before their line; trying gives a clear error. The simple rule that keeps you out of trouble: declare things before you use them, top to bottom.

⑧ Closures — functions that remember Advanced

This one sounds scary but the idea is simple and you've half-seen it already. A closure happens when a function "remembers" the variables from the scope where it was created, even after that outer scope has finished running. The inner function keeps a private link to those variables.

Why care? Closures let you create private state — values that only one function can touch. The classic demo is a "counter maker": a function that builds and returns another function, where each returned counter has its own private count that nobody else can reach or corrupt.

⑨ Higher-order functions & callbacks Intermediate

In JavaScript, functions are values — just like numbers and strings. That means you can store a function in a variable, put it in an array, and (the powerful part) pass a function into another function as an argument. A function that takes another function as input, or returns one as output, is called a higher-order function. The function you pass in is called a callback, because the outer function "calls it back" at the right moment.

This feels abstract until you see it: .map(), .filter(), .forEach(), addEventListener, setTimeout — they're all higher-order functions. You give them a callback and they decide when to run it (for each item, on each click, after a delay).

💡

Passing behaviour around as a value is what makes map/filter/reduce (coming up) so flexible — you supply the "what to do with each item" part, and the method handles the looping. It's also the entire basis of event handling in React: onClick={handleClick} passes your function for React to call back later.

⑩ Arrays — lists of things Basics

An array is an ordered list of values, written between square brackets: ["a", "b", "c"]. Use an array whenever you have "many of the same kind of thing" — a list of names, scores, products, messages. Each item has a position called its index, and here's the part that trips up everyone at first: indexes start at 0, not 1. So the first item is at index 0, the second at index 1, and the last is at index length - 1.

Arrays come with built-in helpers (called methods) you call with a dot: .push() adds to the end, .pop() removes the last, .length tells you the count, .includes() checks for membership. We'll meet the really powerful ones — map, filter, reduce, find, sort — shortly.

⑬ Objects — labelled bundles of data Basics Intermediate

An object groups related values under names (called keys or properties). If an array is a numbered list, an object is a labelled form: instead of item[0] you write user.name. This is how real-world "things" — a user, a product, a task — are represented in code, because real things have named attributes, not numbered positions.

You read a property with dot notation (user.name) or, when the key is dynamic or has special characters, bracket notation (user["name"]). You can add or change a property at any time just by assigning to it: user.city = "Pune". Asking for a property that doesn't exist gives you undefined rather than an error.

⑱ Optional chaining ?. & nullish ?? Intermediate Advanced

These two tiny operators solve a huge real-world pain: dealing with data that might be missing. When you fetch a user from an API, maybe their address didn't come back. Reading user.address.city would then crash with "Cannot read properties of undefined" — one of the most common errors beginners hit.

Optional chaining ?. says "if the thing before me is null or undefined, stop and give back undefined instead of crashing." So user?.address?.city safely returns undefined when anything along the path is missing.

Nullish coalescing ?? means "use the left side, unless it's null or undefined — then use the right." It's like || but only falls back on null/undefined, not on every falsy value. That difference matters: with ||, a valid 0 or "" would wrongly trigger the fallback; with ?? it won't.

⑲ Sets & Maps — specialised collections Intermediate

Arrays and objects cover most needs, but two other collections are worth knowing for the jobs they do better.

A Set is a list that automatically keeps only unique values — duplicates are ignored. The instant use: removing duplicates from an array. Wrap an array in new Set(...) and spread it back out.

A Map is like an object (key → value pairs) but its keys can be any type (not just strings), it remembers insertion order, and it has a clean .size plus .get()/.set()/.has() methods. Reach for a Map when you're building a lookup table or counting things.

⑳ Immutability — why we avoid mutating Intermediate Advanced

To mutate means to change something in place. arr.push(x) mutates the array; user.age = 31 mutates the object. It feels natural, but in modern apps — especially React — we deliberately avoid mutating shared data. Instead we make a copy with the change applied, leaving the original untouched.

Why? Two reasons. First, when several parts of your code share the same object, one piece mutating it can mysteriously break another (a "spooky action at a distance" bug). Second, React decides whether to re-render by checking if an object/array is a new one — if you mutate in place, it's still the same reference, and the screen won't update. So the rule becomes: don't change it, replace it.

Your tools for copying-with-changes are ones you already met: spread ({...obj}, [...arr]) and map/filter (which always return new arrays).

This is one of the most important habits for React. "Never mutate state — return a new copy" will make total sense once you're updating useState values. You're building the muscle memory now.

㉑ The this keyword — a gentle intro Advanced

this is JavaScript's most confusing keyword, so we'll keep it light. In plain terms, this refers to "the object the current function is attached to" — but crucially, what this points to depends on how the function is called, not where it's written. Inside an object method, this usually means "the object this method belongs to."

㉓ Promises — a value that arrives later Intermediate Advanced

Before async/await, you need to meet the thing they're built on: the Promise. A Promise is an object that represents a value that isn't ready yet but will be — like a restaurant buzzer. When you order food, you don't stand frozen at the counter; you get a buzzer (the Promise) and go sit down. The buzzer eventually does one of two things: lights up "your food is ready" (the Promise resolves with a value) or "sorry, we're out" (the Promise rejects with an error).

You attach handlers with .then() (runs when it resolves, receiving the value) and .catch() (runs if it rejects, receiving the error). Many built-in things hand you Promises already — most famously fetch() for calling APIs.

㉔ Async & await — Promises made readable Intermediate Advanced

Some tasks take time: fetching data from a server, reading a file, querying a database. JavaScript doesn't freeze while it waits — it carries on, then comes back when the result is ready. This is asynchronous code, and it's everywhere in real apps. async/await is modern syntax that lets you work with the Promises you just learned, but write them in a clean top-to-bottom style that reads like ordinary step-by-step code.

Two keywords: mark a function async to use await inside it. Then await somePromise means "pause right here until this Promise resolves, then give me its value." Under the hood it's the same Promise from the previous section — await is just unwrapping it for you.

㉕ Error handling — failing gracefully Intermediate Advanced

Things go wrong: a server is down, input is malformed, a file is missing. Without handling, an error crashes the running code. The job of error handling is to catch the problem, respond sensibly (show a message, use a default, log it), and keep the app alive.

When something goes wrong, JavaScript creates an Error object — it has a .name (like "TypeError") and a .message describing the problem. You can throw your own errors too, with throw new Error("..."), to signal "this situation is invalid — stop and handle it." You catch all this with try / catch, and an optional finally block runs no matter what (success or failure) — perfect for cleanup.

㉗ Modules — splitting code across files Intermediate

Real projects aren't one giant file — they're many small files, each focused on one job. Modules are how files share code: one file exports the things it wants to make available, and another file imports them. This keeps code organised and reusable.

There are two flavours of export. A named export exposes specific things by name (you import them inside { }). A default export marks the one main thing a file provides (you import it with any name you like, no braces). Because these examples span multiple files, they're shown as static code rather than runnable here.

math.js — exporting
// Named exports — expose specific values by name
export const PI = 3.14159;
export function add(a, b) {
  return a + b;
}

// Default export — the file's main thing (only one allowed)
export default function multiply(a, b) {
  return a * b;
}
app.js — importing
// Default import (any name) + named imports (in braces)
import multiply, { PI, add } from "./math.js";

console.log(add(2, 3));     // 5
console.log(multiply(4, 5)); // 20
console.log(PI);            // 3.14159
💡

This is the same import/export syntax you'll use in every React component file and most Node code. Node also has an older style (require() / module.exports) — we cover both, and when to use each, in the Node module.

Quiz — check your understanding

1. What is the value of fruits[0] for const fruits = ["a", "b", "c"]?
"a"
"b"
An error
✅ Array indexes start at 0, so fruits[0] is the first item, "a".
2. Which method creates a new array containing only items that pass a test?
.map()
.filter()
.reduce()
✅ .filter() keeps items that return true. .map() transforms every item; .reduce() boils the list down to one value.
3. What does await do?
Stops the entire program forever
Pauses the function until a result is ready, without freezing the app
Makes code run faster
✅ await waits for an async result (like a server response) inside an async function, while the rest of the app keeps running.
4. Why should you use === instead of ==?
It compares value and type, avoiding surprising bugs
It's faster to type
There's no difference
✅ === is strict equality — no surprising type conversions. 0 == "" is true (yikes!), but 0 === "" is false.
5. Which of these is falsy?
"0" (string zero)
[] (empty array)
"" (empty string)
✅ The empty string "" is falsy. Tricky ones: "0" and [] are both truthy — only empty strings, 0, null, undefined, NaN, and false are falsy.
6. What does [10, 2, 1].sort() return (no compare function)?
[1, 2, 10]
[1, 10, 2]
An error
✅ Default .sort() compares items as strings, so "10" sorts before "2". For numbers, pass a compare function: .sort((a, b) => a - b).
7. const u = {}; console.log(u?.profile?.city) prints what?
A crash / TypeError
undefined
null
✅ Optional chaining ?. stops safely and returns undefined instead of crashing when part of the path is missing. Without ?., u.profile.city would throw.
8. To update an object in React-friendly style, you should…
Mutate it directly: user.age = 31
Make a copy with the change: { ...user, age: 31 }
Delete it and start over
✅ Prefer immutability — copy with spread and apply the change. This avoids spooky shared-state bugs and lets React detect the change and re-render.

Key takeaways

  • Store values in variables (const by default, let when it changes); let/const are block-scoped — never use var.
  • The core types: string, number, boolean, array, object, plus null/undefined. Build text with template strings `${...}`.
  • Know the short falsy list (false, 0, "", null, undefined, NaN) — everything else is truthy. Compare with ===.
  • Decide with if/else, the ternary cond ? a : b, and switch.
  • Functions package reusable logic; arrow functions (x) => ... are the modern style and don't bind their own this.
  • Higher-order functions take callbacks; closures let functions remember private state.
  • Master string methods (slice, split, includes, trim…) and Math/Number() for converting and rounding.
  • Arrays of objects model real data; map / filter / reduce / find / sort are your daily power tools (remember the numeric-sort gotcha).
  • Use default & rest params, optional chaining ?. and nullish ?? to handle missing data safely.
  • Prefer immutability — copy with spread/map instead of mutating; it's a core React habit.
  • Promises (.then()/.catch()/Promise.all) and async / await handle anything that takes time; wrap them in try/catch.
  • Recognise destructuring, spread, and import/export — they're everywhere in React and Node.
