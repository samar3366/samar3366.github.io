⚛️ React: Build Interactive UIs

React is a JavaScript library for building user interfaces out of small, reusable pieces called components. The playgrounds below are real React — edit the code and watch the live preview update.

🌐

The live React playgrounds load React & Babel from a CDN, so keep this page online the first time. If a preview stays blank, check your internet connection and click Run.

Why React? The big idea

Updating a page by hand with the DOM (Module 1) gets messy fast. Imagine a shopping cart: when the user adds an item you have to manually find the cart count element and update its text, find the totals element and recalculate it, find the "checkout" button and maybe enable it, and find the empty-cart message and hide it. Miss one of those steps and your UI is now lying to the user. As an app grows, the number of "if this changed, remember to update that" rules explodes — this is the classic source of buggy, fragile web apps.

React flips the model: you describe what the UI should look like for a given state, and React figures out how to update the screen. You stop writing step-by-step DOM instructions ("imperative" code) and instead write a description of the result ("declarative" code). Think of it like the difference between giving someone turn-by-turn driving directions versus just handing them the destination address — React is the GPS that works out the route for you.

🧩

Components: build your UI from small, reusable blocks (a Button, a Card, a TaskList) — like LEGO. Each block owns its own markup, styling, and behavior, so you can build it once and reuse it everywhere.

🔄

State-driven: change the data, and the UI updates itself automatically. No manual DOM juggling, no forgetting to update one corner of the screen — the UI is always a reflection of your current data.

Two more reasons React is worth learning: it's one of the most in-demand skills in web development (the "R" in MERN, the stack this whole course builds toward), and once you understand the core ideas — components, props, state — they carry over to almost every other modern UI framework you'll meet.

🔗

Everything from the JavaScript module pays off here: arrow functions become event handlers, .map() renders lists, destructuring reads props, the spread operator updates state immutably, and template literals build dynamic text. React isn't a new language — it's JavaScript with a smart UI layer on top.

① Components — reusable UI blocks Basic

A component is just a function that returns what to show. That's the whole idea — if you can write a JavaScript function, you can write a component. It takes in some data and returns a description of some UI.

By convention, component names start with a Capital letter (Welcome, not welcome). This isn't a style preference — React uses the capital letter to tell your components apart from built-in HTML tags. A lowercase <welcome /> would be treated as an unknown HTML element and silently render nothing, which is a classic beginner head-scratcher.

A component is reused by writing it like an HTML tag: <Welcome />. You can place it as many times as you like, and each copy is independent.

② JSX — HTML inside JavaScript Basic

That HTML-looking syntax inside the function is JSX. It lets you write markup right in your JavaScript, which feels strange at first but quickly becomes natural — your markup and the logic that drives it live in the same place, instead of being scattered across separate HTML and JS files.

Behind the scenes, JSX isn't really HTML and it isn't magic. A build tool (Babel, which also powers these playgrounds) converts each tag into a plain JavaScript function call that creates a lightweight object describing the element. So <h2>Hi</h2> becomes something like React.createElement("h2", null, "Hi"). You never write that yourself — but knowing JSX is "just JavaScript in disguise" explains why you can use loops, variables, and expressions so freely inside it. A few rules to remember:

  • Return one top-level element (wrap siblings in a <div> or a Fragment <>...</>).
  • Use className instead of class, and htmlFor instead of for (because class and for are reserved JavaScript keywords).
  • Every tag must be closed. Self-closing tags need the slash: <img />, <br />, <input />.
  • Embed any JavaScript expression with curly braces { } — a variable, a calculation, a function call, a ternary. (Statements like if or for don't go inside { } directly.)
jsx.jsx
⚠️

Curly braces hold JavaScript: {year + 1}, {user.toUpperCase()}, {isVip ? "yes" : "no"}. For inline styles, note the double braces: style={{ }} — one set says "JavaScript goes here", the inner set is the style object. Style properties are camelCased (backgroundColor, not background-color) and values are strings or numbers.

💡

Two values render as nothing in JSX, which is handy: null and false. That's the trick behind conditional rendering, which you'll see shortly. But beware: the number 0 does show up, which causes a famous bug we'll cover.

③ Fragments — grouping without extra divs Basic

A component can only return one root element. The lazy fix is to wrap everything in a <div> — but that adds an extra, meaningless box to your page for every component, which can break CSS layouts (flexbox, grids, tables) and bloats the HTML. This is sometimes called "div soup".

The clean solution is a Fragment: an empty tag <>...</> that groups children together without rendering anything itself. It satisfies the "one root" rule but leaves no trace in the actual DOM. Try swapping the <> below for a <div> and inspect the difference:

fragment.jsx
🔑

Reach for a Fragment whenever you need to return multiple elements but a wrapper <div> would get in the way — especially inside tables (<tr>/<td>) and flex/grid layouts. The long form is <React.Fragment>; you only need that when you have to give the fragment a key (e.g. in a list).

④ Props — passing data into components Basic

Props (short for "properties") let a parent pass data into a child component — exactly like passing arguments into a function. This is the mechanism that makes components reusable: one <Greeting /> component can serve many different names, just like one greet(name) function can greet many people.

props.jsx
🔑

Props flow down, parent → child, and are read-only inside the child. A child must never try to modify its own props — if it needs to change something, that something belongs in state (coming up) or should be changed by the parent. One reusable component, many uses — that's the payoff.

💡

You can pass any JavaScript value as a prop — strings, numbers, booleans, arrays, objects, even functions. Strings use quotes (name="Ravi"); everything else uses curly braces (count={5}, tags={["a","b"]}, onClick={handleClick}). Passing a function as a prop is how a child tells its parent "something happened" — you'll use this constantly.

If you write <Greeting name="Mei" />, how do you read name inside the component?
From the props object — e.g. props.name or by destructuring { name }
From a global variable called name
With useState("Mei")
✅ Every prop arrives as a key on the single props object the component receives. Destructuring function Greeting({ name }) is just a tidy shortcut for reading props.name.

⑤ Composition & the children prop Intermediate

Sometimes you want a component to wrap whatever you put inside it — a Card that adds a border and padding around any content, a Modal that frames any message, a Layout that surrounds a page. React has a special built-in prop for exactly this: children. Whatever you place between a component's opening and closing tags shows up as props.children.

⑥ State — data that changes over time Basic

Props come from outside and don't change. State is a component's own private memory that does change over time — the typed text in a search box, whether a menu is open, the current count, the list of todos. When state changes, React automatically re-renders the component so the screen stays in sync with the data.

You create state with the useState hook. A "hook" is just a special function (its name always starts with use) that lets a component plug into React features like memory and lifecycle. useState is the one you'll use most.

Here's the classic counter. Click the buttons — React updates the display every time:

counter.jsx
⚠️

Never change state directly (don't write count = 5). Always use the setter (setCount(5)) — that's what tells React to re-render. Assigning to the variable does nothing visible and just confuses React.

🧠

State updates can be "batched" and are not instant. Right after calling setCount(count + 1), the count variable still holds the old value until the next render. If your new value depends on the old one, use the updater function form: setCount(c => c + 1). React hands you the latest value as c — this is the safe way to do "+1" reliably, even multiple times in a row.

🧪 Try it yourself

Change the + button so it adds 5 at a time, and add a fourth button that doubles the count using the updater form: onClick={() => setCount(c => c * 2)}.

⑦ Events — responding to the user Basic

React events look like HTML attributes but use camelCase and take a function, not a string: onClick, onChange, onSubmit, onMouseEnter, and so on. The function runs when the event fires.

One subtle but important point: you pass the function, you don't call it. Write onClick={handleClick} (pass the function) — not onClick={handleClick()} (which calls it immediately during render and stores the result). When you need to pass arguments, wrap it in an arrow function: onClick={() => setOn(!on)}. That arrow is a tiny function React will call later, on click. Try this live toggle:

events.jsx
⚠️

Pass, don't call. onClick={doThing} ✅ vs onClick={doThing()} ❌. The second runs doThing on every render, often causing an infinite re-render loop if it sets state. When in doubt, wrap it: onClick={() => doThing()}.

⑧ Conditional rendering — showing things only sometimes Intermediate

Real UIs constantly show or hide things: a "Loading…" spinner, an error banner, a logged-in vs logged-out view, an empty-state message. Because JSX is just JavaScript, you render conditionally using ordinary JavaScript expressions. There are three patterns you'll use over and over:

  • Logical AND (&&) — "show this only if the condition is true." {isOpen && <Menu />}. If the left side is true, JSX renders the right side; if false, it renders nothing.
  • Ternary (? :) — "show A if true, otherwise B." {loggedIn ? <Dashboard /> : <Login />}.
  • Early return — bail out of the whole component before the main markup: if (loading) return <p>Loading…</p>;. Great for guard clauses.

Click the buttons below to see all three in action:

conditional.jsx
⚠️

The 0 trap. With {count && <Badge />}, if count is 0, React renders the number 0 on screen instead of nothing! That's because 0 is "falsy" but still renderable. Fix it by making the left side a real boolean: {count > 0 && <Badge />}.

💡

Use early returns to keep components readable: handle loading and error states up top with if (...) return ...;, so your main JSX at the bottom only deals with the happy path. You'll lean on this in the data-fetching section.

⑨ Rendering lists with .map() Intermediate

Remember .map() from the JavaScript module? In React you use it to turn an array of data into an array of elements. This is the bridge between your data (which often comes from a server) and what the user sees on screen. Each item needs a unique key so React can track it across re-renders.

list.jsx
🔑

The key prop (a stable, unique id) is how React keeps track of which item is which between renders. When the list changes, React compares keys to figure out what was added, removed, or reordered — so it can touch only the parts of the DOM that actually changed, instead of rebuilding the whole list.

Why keys matter — and what breaks without them

Imagine a numbered coat check. The key is the ticket number. If everyone has a stable ticket, the attendant instantly finds your coat no matter how the line shuffles. But if tickets are handed out by position in line (the array index), then when someone leaves the front, everyone's ticket number shifts — and the attendant hands you the wrong coat.

That's exactly the bug with index keys on a changing list. Because each item's key (its position) changes when items are inserted, removed, or reordered, React mismatches elements. The classic symptom: you delete one item from a list of inputs, and the text you typed jumps to the wrong row, or a checkbox stays checked on the wrong item. The data is correct, but the on-screen state is glued to the wrong element.

⚡

Rule of thumb: use a real, stable id from your data as the key. Using the array index is only safe when the list never changes order and items are never inserted or removed (a static list). When in doubt, give your items real ids.

Why does React need a key on each list item?
So it can match items across re-renders and update only what changed
Because keys set the CSS order of the items
Keys are optional decoration with no real effect
✅ Keys are React's identity tags for list items. With stable keys, React efficiently adds/removes/reorders the right elements instead of rebuilding the whole list.

⑩ Derived values — don't store what you can compute Intermediate

A super common beginner habit is to create extra state for things that could simply be calculated from existing state. For example, storing both a todos array and a separate completedCount number. The problem: now you have two sources of truth that can drift out of sync — forget to update the count in one place, and your UI shows a wrong number.

The fix is the principle of derived (computed) values: keep the minimal state, and compute everything else during render. The count, the filtered list, whether a form is valid — these are all just expressions over your state. Because the component re-renders whenever state changes, your computed values are always fresh and correct, for free.

derived.jsx
🔑

Click an item — the counts update automatically because they're derived from items, not stored separately. Ask yourself before adding state: "Can I calculate this from data I already have?" If yes, compute it; don't store it.

⑪ Forms — capturing input Intermediate

Forms tie everything together: state holds the input value, onChange updates that state on every keystroke, and onSubmit handles the final result. An input wired this way — value from state, updates state on change — is called a controlled component, because React (not the browser) is the single source of truth for what's in the box. Many fields, one handler — a state object

Most real forms have several fields (name, email, message…). You could create a separate useState for each, but that gets repetitive fast. A tidy pattern is to keep one state object and use a single change handler that updates the right field by its name attribute. The key technique is the spread operator from the JavaScript module: copy the old object, then overwrite one field — { ...form, [name]: value } (a computed property name).

⚠️

Notice we build a new object with { ...form, ... } rather than mutating form directly (no form.name = value). React only re-renders when you hand the setter a brand-new value — mutating the old object in place would update the data but leave the screen stale. Immutability is a recurring theme; the pitfalls section drives it home.

⑫ Lifting state up — sharing between siblings Intermediate

What happens when two components need to share the same data? Say a temperature slider and a "feels like" label both depend on one number. State is private to the component that owns it, so siblings can't reach into each other. The React answer: lift the state up to their nearest common parent, then pass it back down to each child as props.

The parent becomes the single source of truth. It holds the state and passes two things down: the current value (so children can display it) and a function to change it (so children can request updates). This "data down, events up" flow is the backbone of how React apps are wired together.

Here the parent owns temp. The slider changes it; the readout reflects it — instantly in sync, because they share one piece of state:

lifting.jsx
🔑

The parent passed setTemp straight down as the onChange prop — so the child can update the parent's state without ever owning it. That's "events up": children don't change shared state themselves, they call a function the parent gave them.

Two sibling components need to share the same piece of data. Where should that state live?
In their closest common parent, passed down as props
Duplicated independently in each sibling
In a global HTML attribute
✅ Lift the state to the nearest common parent and pass it down. Keeping one source of truth avoids the two copies drifting out of sync.

⑬ useEffect — side effects & cleanup Advanced

Rendering should be "pure" — given the same state, a component returns the same JSX and touches nothing outside itself. But real apps need to reach outside React sometimes: fetch data from a server, start a timer, subscribe to events, set the document title. These are called side effects, and the useEffect hook is where they belong. The function you give useEffect runs after React has painted the screen.

The dependency array controls when it runs

The second argument to useEffect — the array — is the most important and most misunderstood part. It tells React which values to "watch". After each render, React compares the array to last time; if anything changed, it runs your effect again. Three cases:

  • useEffect(fn, []) — empty array: run once after the first render (on "mount"). Perfect for fetching initial data.
  • useEffect(fn, [count]) — with dependencies: run on mount and every time count changes.
  • useEffect(fn) — no array at all: run after every render. Rarely what you want, and a common cause of infinite loops.

Cleanup — the return function

If your effect starts something that keeps running — a timer, a subscription, a WebSocket — you must stop it when the component goes away (or before the effect re-runs). Otherwise you get "zombie" timers stacking up, memory leaks, and updates fired at components that no longer exist. To clean up, return a function from your effect; React calls it on unmount and before each re-run. The interval below is created on mount and cleared on cleanup — toggle the clock off and the timer stops cleanly:

⚠️

Without cleanup, that setInterval would keep ticking forever, even after the clock is hidden — and if you showed/hid it repeatedly you'd pile up multiple timers all fighting to update state. Returning clearInterval(id) tears it down. Same idea for removeEventListener, unsubscribing, and aborting fetches.

Fetching data — the original use case

The most common use of useEffect is loading data from a server after the component appears. The empty [] means "fetch once on load":

fetch-pattern.jsx
import { useState, useEffect } from "react";

function Users() {
  const [users, setUsers] = useState([]);   // start empty
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // runs once, after first render
    async function load() {
      const res = await fetch("http://localhost:4000/api/users");
      const data = await res.json();
      setUsers(data);       // store it → triggers re-render
      setLoading(false);
    }
    load();
  }, []);                    // [] = run once

  if (loading) return <p>Loading…</p>;
  return (
    <ul>
      {users.map((u) => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}
🔑

This exact pattern — useState for the data + useEffect to fetch it on load — is how your React front-end will talk to the Node/Express REST API you build in Modules 3–5. You'll wire it up for real in the capstone. See the data-fetching states section below for the full loading/error/empty handling.

When does useEffect(fn, [userId]) run its effect?
After the first render, and again whenever userId changes
Only once, ever — the dependency is ignored
Before the component renders, blocking the paint
✅ React re-runs the effect whenever a value in the dependency array changes. So an effect that fetches a user should list [userId] to re-fetch when you switch users.

⑭ useRef — DOM access & values that don't re-render Advanced

Sometimes you need to do something that state can't elegantly handle. The useRef hook gives you a small, persistent "box" — an object with a single .current property — that survives re-renders. It has two main jobs:

  • Grab a real DOM element — to focus an input, scroll to a spot, measure a size, or play a video. Attach a ref to a tag with ref={myRef}, then reach the element via myRef.current.
  • Hold a mutable value that should NOT trigger a re-render — like a timer id or a "previous value". Changing ref.current never re-renders the component (unlike state).

Below, a ref points at the input so the button can call .focus() on the actual DOM node:

💡

State vs ref, in one line: use state for data that should appear on screen (changing it re-renders); use a ref for behind-the-scenes values or direct DOM access (changing it doesn't re-render). Reach for refs only when the declarative, state-driven approach genuinely can't do the job — they're the escape hatch, not the default.

⑮ Custom hooks — reusing stateful logic Advanced

⑯ Context API — escaping prop-drilling Advanced

Passing props down one level is easy. But what if a deeply nested component needs data that lives way up at the top — the logged-in user, the color theme, the language? Threading that prop through every intermediate component that doesn't even use it is tedious and fragile. This pain has a name: prop-drilling.

The Context API solves this. You createContext once, wrap part of your tree in a <Context.Provider value={...}>, and any component inside — no matter how deep — reads the value with useContext. No more passing props through the middle. (This needs real imports, so here's a static example rather than a playground.)

theme-context.jsx
import { createContext, useContext, useState } from "react";

// 1. Create the context (with a default)
const ThemeContext = createContext("light");

// 2. Provide a value high in the tree
function App() {
  const [theme, setTheme] = useState("dark");
  return (
    <ThemeContext.Provider value={theme}>
      <button onClick={() => setTheme(theme === "dark" ? "light" : "dark")}>
        Toggle theme
      </button>
      <Toolbar />          {/* doesn't need the theme prop */}
    </ThemeContext.Provider>
  );
}

function Toolbar() {
  return <SaveButton />;   // also doesn't pass anything down
}

// 3. Consume it anywhere below, no prop-drilling
function SaveButton() {
  const theme = useContext(ThemeContext);   // reads "dark"
  return <button className={theme}>Save ({theme})</button>;
}
⚡

Context is perfect for global-ish data that many components need: theme, current user, language, auth status. Don't overuse it for everything though — for most local data, plain props and lifting state up are simpler and clearer.

⑰ Data fetching: loading, error & empty states Advanced

Fetching data isn't just "get the array and map it." Networks are slow and unreliable, so a polished component handles four distinct states: still loading, request failed, succeeded but empty, and succeeded with data. Skipping these is the difference between an amateur UI (blank screen, then a flash of content) and a professional one.

The recipe: keep three pieces of state — the data, a loading flag, and an error — then use early returns (from the conditional-rendering section) to render the right thing for each case. Because this needs a real server and fetch, it's a static block; you'll run it against the API from Module 5 (REST APIs) in the capstone.

UserList.jsx
import { useState, useEffect } from "react";

function UserList() {
  const [users, setUsers]     = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  useEffect(() => {
    let active = true;                         // avoid setting state after unmount
    async function load() {
      try {
        setLoading(true);
        const res = await fetch("http://localhost:4000/api/users");
        if (!res.ok) throw new Error("Server returned " + res.status);
        const data = await res.json();
        if (active) setUsers(data);
      } catch (err) {
        if (active) setError(err.message);     // network or server error
      } finally {
        if (active) setLoading(false);
      }
    }
    load();
    return () => { active = false; };           // cleanup guards against late updates
  }, []);

  // 1. LOADING state
  if (loading) return <p>⏳ Loading users…</p>;
  // 2. ERROR state
  if (error)   return <p style={{ color: "crimson" }}>⚠️ {error}</p>;
  // 3. EMPTY state
  if (users.length === 0) return <p>No users yet — add one!</p>;
  // 4. SUCCESS state
  return (
    <ul>
      {users.map((u) => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}
🔑

Always handle all four states. The try/catch/finally guarantees loading turns off even if the request fails, and the active flag in the cleanup prevents the "can't update state on an unmounted component" warning when a user navigates away mid-request.

Thinking in React — breaking a UI into components Intermediate

Before you write a single line, it helps to look at a design and mentally chop it into a tree of components. The official React mindset is a few simple steps: draw boxes around every logical piece of the UI, nest them into a hierarchy, decide what's static (props) versus what changes (state), figure out the minimal state and where it should live, and wire the data flow.

Imagine a product page with a search box, a filter checkbox, and a list of products grouped by category. You might break it into: a FilterableProductTable wrapping a SearchBar and a ProductTable, where the table renders ProductCategoryRows and ProductRows. Each box is one component with one job.

⚡

Two handy questions when sizing components: "Does this piece have one clear job?" (if it has several, split it) and "Is this exact markup repeated?" (if so, extract a reusable component with props). You'll do this for real when you build the capstone's task board.

Common pitfalls & the Rules of Hooks Advanced

Almost every React bug a beginner hits falls into a handful of categories. Learn to recognize these and you'll save hours:

🚫

Mutating state directly. items.push(x) or form.name = "x" changes the old object/array in place. React sees the same reference and skips re-rendering. Always make a new value: setItems([...items, x]), setForm({ ...form, name: "x" }).

⏳

Stale state. State variables hold the value from this render. setCount(count + 1) twice in a row only adds 1, because both read the same old count. Use the updater form: setCount(c => c + 1).

🔑

Missing or index keys. No key triggers a warning and slow updates; index keys on a reorderable list cause UI state to stick to the wrong row. Use stable ids.

🪝

Breaking the Rules of Hooks. Call hooks only at the top level of a component — never inside if, loops, or nested functions — and only from React components or other hooks. The order of hook calls must be identical every render.

That last one trips people up, so here's the why: React tracks each hook by the order it's called. If a hook is hidden behind an if, the order changes between renders and React loses track of which state is which. The fix is always to put the condition inside the hook, not the hook inside the condition.

❌ Wrong
function Bad({ show }) {
  if (show) {
    // ❌ hook inside a condition
    const [n, setN] = useState(0);
  }
  // ...
}
✅ Right
function Good({ show }) {
  // ✅ hook always called, top level
  const [n, setN] = useState(0);
  if (!show) return null;   // condition AFTER hooks
  // ...
}
Why must you never call a hook inside an if statement?
React identifies hooks by call order, which must be the same every render
Hooks are too slow to run conditionally
It's only a style preference with no real consequence
✅ React relies on hooks being called in the same order each render to match up state. A conditional hook changes that order and corrupts the state mapping — hence the Rules of Hooks.
Which line correctly adds an item to a state array items without mutating it?
items.push(newItem)
setItems([...items, newItem])
items = [...items, newItem]
✅ Spread the old array into a brand-new one and pass it to the setter. push mutates in place (no re-render); reassigning the variable does nothing useful in React.

Key takeaways

  • Components are functions that return JSX — your reusable UI blocks. Fragments (<>…</>) group siblings without an extra div.
  • JSX is HTML inside JavaScript; embed values with { }, use className. null/false render nothing — the basis of conditional UI.
  • Props pass data down (read-only); the children prop enables composition. State (useState) is changeable memory that re-renders on change.
  • Conditional rendering uses &&, ternaries, and early returns. Render lists with .map() + a stable key (not the index).
  • Prefer derived values over extra state. For shared data, lift state up to a common parent ("data down, events up").
  • Handle events by passing functions (not calling them). Build forms as controlled inputs, often with one state object + one handler.
  • useEffect runs side effects; the dependency array controls when, and the returned cleanup function stops timers/subscriptions.
  • useRef grabs DOM nodes / holds non-rendering values. Custom hooks (useX) reuse stateful logic; Context avoids prop-drilling.
  • Fetching needs loading / error / empty / success states. Avoid pitfalls: don't mutate state, beware stale state, and obey the Rules of Hooks.
  • Start real projects with Vite (npm create vite@latest).
