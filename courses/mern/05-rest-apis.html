🔌 REST APIs: The Contract

An API is how your React front-end and Node back-end talk to each other. REST is the most common set of rules for designing that conversation. Master it and you can connect any front-end to any back-end — and, just as importantly, understand every web app you'll ever build or use.

🗺️

Where you are: you just finished the Node module, so you can already build a server and write routes. This module is the bridge between that server and the React app you built earlier. Next module, you'll swap our temporary in-memory data for a real MSSQL database — but the API shape you learn here won't change. That's the whole point of a good contract.

What is a REST API?

API = Application Programming Interface — a defined way for two programs to talk. A REST API organises that conversation around resources (things like "users", "tasks", "products"), each with a URL, acted on by standard HTTP methods. "REST" stands for Representational State Transfer — a fancy name for a simple idea: the client and server exchange representations (usually JSON) of resources, using the plain verbs that the web (HTTP) already gives us.

Why does REST matter? Because it's a shared convention. If everyone agrees that GET /api/tasks means "give me the tasks" and DELETE /api/tasks/5 means "delete task 5", then any front-end can talk to any back-end without a custom instruction manual. You're not inventing a new language for every project — you're speaking one the whole industry already understands.

🔑

An API is a contract. The front-end promises to send requests in an agreed shape; the back-end promises to respond in an agreed shape. As long as both sides honour the contract, either side can be rewritten, replaced, or scaled independently. That separation is the superpower of full-stack development.

① Anatomy of an HTTP request & response Basic

Before we go deeper into methods and status codes, let's slow right down and look at what a single request and its response are actually made of. Every time your browser loads a page or your React app calls fetch, this exact structure flies across the network as plain text. Once you can picture these pieces, nothing about APIs will feel mysterious again.

The request — three (sometimes four) parts

A request is a small text message that asks the server to do something. It has up to four parts, in this order:

  1. Request line — the method, the path, and the HTTP version. Example: POST /api/tasks HTTP/1.1. This is the "what do you want and where" line.
  2. Headers — a list of Key: Value lines giving metadata about the request (what format the body is in, who's asking, what they can accept back). More on these in the next section.
  3. A blank line — separates the headers from the body. Small but mandatory.
  4. Body (optional) — the actual data you're sending, usually JSON. GET and DELETE requests typically have no body; POST and PUT almost always do.
a raw HTTP request
POST /api/tasks HTTP/1.1          <-- request line: method + path + version
Host: localhost:4000             <-- headers start here
Content-Type: application/json    <-- "my body is JSON"
Accept: application/json          <-- "please reply with JSON"
Authorization: Bearer eyJhbGc...  <-- "here's my login token"
                                  <-- BLANK LINE separates headers from body
{ "title": "Buy milk" }           <-- the body (the data)

The response — the same shape, mirrored

The server replies with a message built almost identically, except the first line is a status line instead of a request line:

  1. Status line — the HTTP version, the status code, and a short reason phrase. Example: HTTP/1.1 201 Created.
  2. Headers — metadata about the response (what format the body is, how long it is, caching rules, etc.).
  3. A blank line.
  4. Body (optional) — the data the server is sending back, usually JSON.
a raw HTTP response
HTTP/1.1 201 Created              <-- status line: version + code + reason
Content-Type: application/json    <-- "my body is JSON"
Content-Length: 41                <-- "the body is 41 bytes"
                                  <-- BLANK LINE
{ "id": 5, "title": "Buy milk", "done": false }   <-- the body
🛠️

You normally never type raw HTTP like this — fetch, Express, and your browser build and parse it for you. But knowing the shape means you can read the "Network" tab in your browser's dev tools and instantly understand what your app is actually sending.

② HTTP headers — the metadata Basic

Headers are Key: Value lines that travel with every request and response. They don't carry the main data (that's the body) — they carry information about the message: what format it's in, who's sending it, what the sender can accept back, how it should be cached, and more. Think of them as the label on a parcel versus the contents inside.

You'll meet a handful constantly. Here are the ones that matter most early on:

HeaderOn request / responseWhat it means
Content-TypeBothThe format of this message's body. application/json for JSON, text/html for a web page, etc. The receiver uses it to know how to parse the body.
AcceptRequestThe format the client would like back. Accept: application/json says "please reply with JSON." It's a preference, not a command.
AuthorizationRequestProof of who you are — usually a token: Authorization: Bearer <token>. The server checks it to decide if you're allowed.
User-AgentRequestIdentifies the client software (which browser/app/version). Set automatically by the browser.
Content-LengthBothHow many bytes the body is. Set automatically.
Cache-ControlResponseTells the browser whether/how long it may reuse this response instead of asking again.
Set-CookieResponseAsks the browser to store a cookie (used for sessions). The browser sends it back on later requests.
🔑

Content-Type vs Accept — the classic confusion. Content-Type describes the body that this very message is carrying. Accept describes the body the client hopes to receive in reply. A POST request often sets both: "I'm sending JSON (Content-Type), and I'd like JSON back (Accept)."

This is exactly why, when you POST from React, you write headers: { "Content-Type": "application/json" } — you're telling Express "the body I'm about to send is JSON, so parse it as JSON." Forget that header and express.json() won't parse your body, and req.body will be empty. A very common beginner bug!

③ HTTP methods — the verbs Basic

Each method expresses an intent. Together they map perfectly to the things you do with data — CRUD: Create, Read, Update, Delete. The method is a promise about what you want to do; the server then decides whether to allow it.

MethodIntentCRUDExample
GETRead / fetch dataReadGET /api/tasks → all tasks
POSTCreate something newCreatePOST /api/tasks → add a task
PUTReplace an existing item (whole)UpdatePUT /api/tasks/5 → replace task 5
PATCHPartially update an itemUpdatePATCH /api/tasks/5 → tweak one field
DELETERemove an itemDeleteDELETE /api/tasks/5 → delete task 5
🔑

The URL names the thing (the resource); the method names the action. /api/tasks/5 always refers to task #5 — what you do to it depends on the method.

PUT vs PATCH — full replace vs partial update

This trips up almost everyone at first, so let's be crystal clear. Both update an existing resource, but they mean different things:

  • PUT = "here is the complete new version, replace what's there." You send all the fields. Any field you leave out may be wiped or reset to a default. Think "overwrite the whole record."
  • PATCH = "here are just the fields I want to change." You send only the changed fields; everything else stays as it was. Think "edit these few things."
put-vs-patch (bodies)
// Existing task on the server:
// { "id": 5, "title": "Buy milk", "done": false, "priority": "low" }

// PUT /api/tasks/5  — must send the WHOLE object (it replaces everything)
{ "title": "Buy oat milk", "done": true, "priority": "high" }

// PATCH /api/tasks/5 — send ONLY what changes (everything else untouched)
{ "done": true }
💡

In practice many beginners (and plenty of real APIs) use PUT loosely to mean "update", merging fields like PATCH. That works, but if you want to be precise and friendly to other developers: use PATCH for partial edits. It signals your intent clearly.

④ Idempotency & safety — two words worth knowing Intermediate

These sound intimidating but the ideas are simple, and understanding them will make you reason about APIs like a pro.

  • Safe = the request doesn't change anything on the server. It only reads. GET is safe — fetching a task ten times changes nothing.
  • Idempotent = making the same request many times has the same end result as making it once. The server might do work each time, but the final state is identical.

Why care? Networks are unreliable. If a response gets lost, a client (or a phone on a flaky connection) might retry. If the method is idempotent, retrying is safe. If it isn't — like POST — a retry might create a second task by accident. That's the real-world "I clicked submit twice and got charged twice" bug.

MethodSafe?Idempotent?Why
GET✅ Yes✅ YesOnly reads; nothing changes no matter how often you call it.
POST❌ No❌ NoCreates a new resource each time — call it twice, get two items.
PUT❌ No✅ YesReplaces with the same data → same result every time.
PATCH❌ No⚠️ Usually*Depends on the change. "Set done=true" is idempotent; "increment count" is not.
DELETE❌ No✅ YesDelete task 5 once or five times → it's gone either way.
⚠️

The one to remember: POST is the odd one out — not safe, not idempotent. That's exactly why double-clicking a "Buy" or "Submit" button can create duplicates, and why such buttons should disable themselves after the first click.

⑤ Status codes — the server's reply code Basic

Every response includes a 3-digit status code summarising what happened. The first digit tells you the family — that alone gets you 80% of the way. Memorise the families first, then learn the common specific codes:

2xx — Success ✅

200 OK · general success
201 Created · new item made (after POST)
204 No Content · success, nothing to return (after DELETE)

3xx — Redirect ↪️

301/302 · the resource lives elsewhere now. You'll rarely set these by hand early on.

4xx — You messed up 🙋

400 Bad Request · invalid data sent
401 Unauthorized · not logged in
403 Forbidden · logged in, not allowed
404 Not Found · no such resource

5xx — Server messed up 💥

500 Internal Server Error · your code threw an error. When you see this, check your server's terminal logs.

🧠

Memory hook: 4xx = client's fault (the request was wrong), 5xx = server's fault (the code broke).

A fuller reference — the codes you'll actually meet

You don't need to memorise all of these, but knowing what each means lets you debug instantly when you see one. Bookmark this table.

400 vs 422: use 400 when the request is broken at the format level (not even valid JSON, missing a required field). Use 422 when the JSON parsed fine but the values don't make sense (email isn't a valid email, price is negative). Both are fine for forms; 422 is just more specific.

⑦ Query parameters — filtering, sorting, paging Intermediate

Sometimes GET /api/tasks would return thousands of items, or you only want some of them. Instead of inventing new URLs, you tack query parameters onto the end after a ?. They're key=value pairs joined by &, and they tweak how a GET behaves without changing which resource you're asking about.

query string anatomy
GET /api/tasks?status=active&sort=-createdAt&page=2&limit=20
//             |---------------- the query string ----------------|
//  status=active   -> filter: only active tasks
//  sort=-createdAt -> sort by createdAt, the "-" means descending
//  page=2          -> pagination: which page of results
//  limit=20        -> how many per page

Three jobs query params do constantly:

  • Filtering — narrow the list: ?status=active, ?done=true.
  • Sorting — order the list: ?sort=title or ?sort=-createdAt (newest first).
  • Pagination — return a slice, not everything: ?page=2&limit=20.

In Express you read these from req.query — already parsed into an object for you (req.query.status, req.query.page). Building them by hand in JavaScript is easy too — the browser gives you URLSearchParams, which handles the fiddly bits like joining with & and escaping special characters:

⑧ Validation — never trust the client Intermediate

This is one of the most important habits in all of back-end development, so let it sink in: anything that reaches your server from the outside is untrusted. A friendly user might forget a field. A buggy front-end might send the wrong type. A malicious actor will deliberately send garbage to break or abuse your API. Front-end validation (a "required" field in a form) is for user convenience only — it's trivially bypassed. The server is the only place validation actually protects you.

Validating means checking, before you act on the data, that it's present, the right type, and within sensible limits — and returning a clear 400 or 422 with a helpful message when it isn't.

validating in a route
router.post("/", (req, res) => {
  const { title, priority } = req.body;

  // 1. Required?
  if (!title || typeof title !== "string" || title.trim() === "") {
    return res.status(422).json({
      error: { message: "title is required", code: "VALIDATION", field: "title" }
    });
  }
  // 2. Within limits?
  if (title.length > 100) {
    return res.status(422).json({
      error: { message: "title must be 100 chars or fewer", code: "VALIDATION", field: "title" }
    });
  }
  // 3. Allowed value?
  const allowed = ["low", "medium", "high"];
  if (priority && !allowed.includes(priority)) {
    return res.status(422).json({
      error: { message: "priority must be low, medium, or high", code: "VALIDATION", field: "priority" }
    });
  }

  // Only now is it safe to use the data:
  const task = { id: nextId++, title: title.trim(), priority: priority || "low", done: false };
  tasks.push(task);
  res.status(201).json(task);
});
💡

Writing these checks by hand teaches you the idea, but in real projects you'll reach for a validation library like Zod or Joi. They let you declare the expected shape once and reject anything that doesn't match, with good error messages for free. Something to explore after this course.

🔑

Golden rule: validate on the server even if you already validate on the client. Client checks improve the experience; server checks keep your data and your app safe. You need both, but the server one is non-negotiable.

⑩ Full CRUD in Express Intermediate

Here's a complete REST API for "tasks" — all four operations. This is the back-end shape of your capstone (we'll swap the in-memory array for MSSQL in the next module). Read it slowly; you already know every piece.

routes/tasks.js
import express from "express";
const router = express.Router();

// Temporary in-memory data (replaced by MSSQL in Module 5)
let tasks = [
  { id: 1, title: "Learn REST", done: true },
  { id: 2, title: "Build API", done: false },
];
let nextId = 3;

// READ all  →  GET /api/tasks
router.get("/", (req, res) => {
  res.json(tasks);
});

// READ one  →  GET /api/tasks/:id
router.get("/:id", (req, res) => {
  const task = tasks.find((t) => t.id === Number(req.params.id));
  if (!task) return res.status(404).json({ error: "Not found" });
  res.json(task);
});

// CREATE  →  POST /api/tasks
router.post("/", (req, res) => {
  const { title } = req.body;
  if (!title) return res.status(400).json({ error: "title is required" });
  const task = { id: nextId++, title, done: false };
  tasks.push(task);
  res.status(201).json(task);     // 201 = Created
});

// UPDATE  →  PUT /api/tasks/:id
router.put("/:id", (req, res) => {
  const task = tasks.find((t) => t.id === Number(req.params.id));
  if (!task) return res.status(404).json({ error: "Not found" });
  task.title = req.body.title ?? task.title;
  task.done = req.body.done ?? task.done;
  res.json(task);
});

// DELETE  →  DELETE /api/tasks/:id
router.delete("/:id", (req, res) => {
  tasks = tasks.filter((t) => t.id !== Number(req.params.id));
  res.status(204).end();          // 204 = success, no content
});

export default router;
server.js — wire it up
import express from "express";
import cors from "cors";
import tasksRouter from "./routes/tasks.js";

const app = express();
app.use(cors());            // allow the React app to call us
app.use(express.json());    // parse JSON bodies
app.use("/api/tasks", tasksRouter);

app.listen(4000, () => console.log("API on http://localhost:4000"));
🌐

CORS: browsers block a page on one address (your React app at :5173) from calling a server on another (:4000) unless the server opts in. The cors() middleware does that. Forget it and you'll see a confusing "CORS error" — there's a whole section on why this happens below.

⑪ Connecting React to your API Intermediate

Back in React, you call your API with the browser's built-in fetch, inside useEffect (for loading) or an event handler (for actions). This is the moment the front-end and back-end become one app:

TaskApp.jsx (front-end side)
const API = "http://localhost:4000/api/tasks";

// READ — load tasks when the component appears
useEffect(() => {
  fetch(API)
    .then((res) => res.json())
    .then((data) => setTasks(data));
}, []);

// CREATE — add a task on form submit
async function addTask(title) {
  const res = await fetch(API, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ title }),
  });
  const newTask = await res.json();
  setTasks((prev) => [...prev, newTask]);   // update UI
}

// DELETE — remove a task
async function removeTask(id) {
  await fetch(`${API}/${id}`, { method: "DELETE" });
  setTasks((prev) => prev.filter((t) => t.id !== id));
}
🔑

See how everything connects? fetch + method + JSON body on the front-end maps directly to app.get/post/delete on the back-end. The status code tells React whether it worked. This is full-stack.

⑫ Authentication & authorization Advanced

📌

This is a concepts section — a map of what to learn next, not a full implementation. You don't need to build auth for this course's capstone, but every real API needs it, so understanding the vocabulary now will pay off enormously.

Two words that sound alike but mean different things:

  • Authentication (authn) — "who are you?" Proving identity. Logging in with email + password is authentication.
  • Authorization (authz) — "what are you allowed to do?" Checking permissions. An admin can delete any task; a normal user can only delete their own. That's authorization.

Tie this back to status codes: 401 Unauthorized means "I don't know who you are" (authentication failed), while 403 Forbidden means "I know who you are, but you can't do this" (authorization failed).

Security essentials — even at beginner level, internalise these:

⑬ CORS explained — why the browser blocks you Advanced

You already met the cors() one-liner. Now let's understand why it's needed, because "CORS error" is the single most confusing message new full-stack developers hit.

The same-origin policy

For security, browsers enforce the same-origin policy: by default, JavaScript running on one origin may not read responses from a different origin. An origin is the combination of protocol + host + port — so http://localhost:5173 and http://localhost:4000 are different origins (different port), even though both are "localhost."

Why does this rule exist? Imagine you're logged into your bank in one tab. Without the same-origin policy, a malicious site in another tab could quietly fire requests to your bank's API using your logged-in session and read the responses. The policy stops that by default. CORS is the controlled, opt-in way for a server to say "actually, this specific other origin is allowed to read my responses."

How the server opts in

CORS works through response headers. When the server wants to allow a cross-origin caller, it adds a header like Access-Control-Allow-Origin: http://localhost:5173. The browser sees that header and permits your JavaScript to read the response. No header → the browser blocks it and logs a CORS error. Crucially, CORS is enforced by the browser, configured by the server — your front-end code can't "fix" it; the fix lives on the back-end.

configuring CORS in Express
import cors from "cors";

// Wide open (fine for learning / public read-only APIs):
app.use(cors());

// Locked down to just your front-end (better for real apps):
app.use(cors({
  origin: "http://localhost:5173",
  methods: ["GET", "POST", "PUT", "PATCH", "DELETE"],
  allowedHeaders: ["Content-Type", "Authorization"],
}));

Preflight requests (the secret extra request)

For "non-simple" requests — anything with a JSON body, a PUT/DELETE/PATCH, or an Authorization header — the browser does something clever first. Before sending your actual request, it sends an automatic OPTIONS request called a preflight, essentially asking "Hey server, am I allowed to send this?" If the server answers with the right CORS headers, the browser then sends the real request. If not, your real request never even leaves the browser.

💡

Good news: the cors() middleware handles preflight OPTIONS requests for you automatically. You rarely write this logic by hand — but when a CORS error appears, you now know it's a server configuration issue, not a bug in your fetch call.

⑭ Good API design — the conventions Advanced

None of these rules are enforced by the computer — your code runs fine either way. They're conventions that make your API predictable to other developers (including future-you). Following them is the difference between an API people enjoy using and one they dread.

Do ✅Avoid ❌
GET /api/tasks (plural nouns)GET /api/getTasks (verb in URL)
DELETE /api/tasks/5GET /api/deleteTask?id=5
Return 201 after creatingAlways returning 200
Nest related: /api/users/3/tasksFlat, ambiguous URLs
Validate input, return 400/422 on bad dataTrusting whatever the client sends

The big ideas behind those rules

  • Resources are nouns, methods are verbs. The URL should name a thing (/tasks), never an action (/getTasks). The action is already expressed by the HTTP method. So GET /tasks, not /getTasks; DELETE /tasks/5, not /deleteTask.
  • Use plural nouns consistently. /tasks for the collection, /tasks/5 for one item in it. Pick plural and stick with it everywhere — mixing /task and /tasks is a constant source of bugs.
  • Nest to show relationships. "The tasks belonging to user 3" reads naturally as /api/users/3/tasks. But don't over-nest — more than two levels deep gets unwieldy; switch to query params instead.
  • Be predictable. If /tasks behaves a certain way, /users and /projects should behave the same way. Consistency means a developer who learns one endpoint already understands the rest.

Statelessness — what it means and why it matters

The "ST" in REST hints at it, but the key principle is that REST APIs are stateless: each request must contain everything the server needs to handle it. The server doesn't remember anything about you between requests — no "we were in the middle of something." If a request needs to know who you are, it carries that proof itself (the Authorization token), every single time.

Why is that a good thing? Because it makes your API scalable and reliable. If the server keeps no per-client memory, you can run ten copies of it behind a load balancer and it doesn't matter which copy handles your next request — they're all interchangeable. If a server crashes and restarts, no conversation is lost, because there was no conversation to lose. Each request stands alone.

🔑

Stateless doesn't mean "no data is stored" — your tasks still live in the database. It means the server process holds no memory of previous requests from a given client. Identity and context ride along in each request, not in the server's head.

💡

Test your API without a front-end using a tool like Postman, the VS Code "REST Client" extension, or curl — see the testing section for hands-on examples.

⑮ Versioning & rate limiting Advanced

API versioning — /api/v1/...

Once real apps depend on your API, you can't freely change it — a change that breaks the response shape would break every client overnight. The solution is to put a version number in the URL: /api/v1/tasks. When you need to make breaking changes, you publish /api/v2/tasks and leave v1 running. Old clients keep working on v1; new clients move to v2 at their own pace.

versioned routes
app.use("/api/v1/tasks", tasksRouterV1);   // the original API
app.use("/api/v2/tasks", tasksRouterV2);   // new shape, old one still works

// Clients choose which they target:
//   GET /api/v1/tasks   -> stable, unchanged forever
//   GET /api/v2/tasks   -> new features / new response shape
💡

You don't need versioning on day one of a hobby project. But knowing the convention means that when you see /v1/ in real APIs, you understand it's a promise: "this shape won't break under you."

Rate limiting — protecting your API

Without limits, one buggy script (or one bad actor) could hammer your API with thousands of requests a second, slowing it to a crawl for everyone or running up your bills. Rate limiting caps how many requests a given client may make in a window of time — say 100 requests per minute. Exceed it and the server replies 429 Too Many Requests, often with a Retry-After header telling you when to try again.

🛡️

In Express you'd add this with a middleware like express-rate-limit — a few lines that track requests per IP and reject the excess. The takeaway for now: APIs protect themselves, and 429 means "you're going too fast, back off."

⑯ Testing & exploring APIs Intermediate

You don't need a finished React front-end to test your back-end — and you shouldn't wait for one. Build and test your API in isolation first. Three tools make this easy:

  • Postman — a friendly app with buttons and forms. Great for poking at APIs visually and saving collections of requests.
  • VS Code REST Client — an extension where you write requests in a .http file right next to your code and click "Send." No leaving the editor.
  • curl — a command-line tool built into macOS/Linux (and modern Windows). No install, scriptable, and great to understand because it maps so directly onto the raw HTTP you saw earlier.

Here are the three workhorse curl commands. Try them against your own Express server once it's running:

curl — GET (read)
# Read all tasks (GET is the default, so no -X needed)
curl http://localhost:4000/api/tasks

# Read one task
curl http://localhost:4000/api/tasks/1

# Add -i to also see the status line and headers in the reply
curl -i http://localhost:4000/api/tasks/1
curl — POST (create, with JSON body)
# Create a task. -X sets the method, -H sets a header, -d sends the body.
curl -X POST http://localhost:4000/api/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"Walk the dog"}'

# Notice the two pieces that MUST match what Express expects:
#   -H "Content-Type: application/json"  -> so express.json() parses it
#   -d '{...}'                           -> the JSON body itself
curl — DELETE, and sending a token
# Delete task 2
curl -X DELETE http://localhost:4000/api/tasks/2

# A protected request, attaching a bearer token:
curl http://localhost:4000/api/tasks \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.xxx.yyy"
💡

See how curl mirrors everything you learned? -X is the method, -H adds headers, -d is the body. It's the raw HTTP request from section ① made typeable. Master this and you can test any API anywhere.

⑰ REST vs GraphQL vs gRPC Advanced

REST is the most common API style, but not the only one. You don't need these for the course — this is just so the names don't surprise you later.

🔭

REST — resources at URLs, standard HTTP verbs, usually JSON. Simple, universal, easy to cache and debug. The right default, and what this whole module teaches.

GraphQL — one endpoint where the client sends a query describing exactly the fields it wants. Great when different screens need different slices of data and you want to avoid over- or under-fetching. More setup to learn.

gRPC — a fast, binary protocol built for service-to-service communication inside large systems. Very efficient, but not something a browser talks to directly. You'll meet it in big back-end architectures.

🔑

Start with REST. It's the lingua franca of the web, it's what 90% of jobs and tutorials use, and the concepts (methods, status codes, headers, JSON) transfer to every other style. Learn the others later, when a real need appears.

Quiz

1. Which method would you use to create a new task?
GET
POST
DELETE
✅ POST creates new resources. GET reads, PUT updates, DELETE removes.
2. You get a 404 response. What does it mean?
The resource wasn't found
The server crashed
Success
✅ 404 Not Found — there's no resource at that URL. It's a 4xx, so it's a client-side issue (wrong URL/id).
3. A 500 error usually means…
You sent invalid data
The server's code threw an error — check its logs
You need to log in
✅ 5xx = server's fault. Something in your back-end code broke; the terminal running your server will show the error.
4. What turns a JavaScript object into text you can send over the network?
JSON.parse()
JSON.stringify()
fetch()
✅ JSON.stringify(obj) → text (to send). JSON.parse(text) → object (after receiving).
5. You only want to change a task's done field, leaving everything else alone. Which method best expresses that?
PUT (send the whole object)
PATCH (send only the changed field)
POST
✅ PATCH = partial update; send just { "done": true }. PUT means "replace the whole resource," so it expects all fields.
6. Which header tells the server "the body I'm sending is JSON"?
Accept
Content-Type
Authorization
✅ Content-Type: application/json describes THIS message's body. Accept describes what you want back; Authorization carries your token.
7. You see a "CORS error" in the browser console. Where is the fix?
In your React fetch call
On the server — it must send the right CORS headers
In the browser settings
✅ CORS is enforced by the browser but configured by the server. You allow the origin server-side (e.g. the cors() middleware in Express).
8. A user is logged in but tries to delete someone else's data. The best status code is…
401 Unauthorized
403 Forbidden
404 Not Found
✅ 403 = "I know who you are, but you're not allowed." 401 means "I don't know who you are" (not authenticated).
9. Why validate input on the server even if the form already checks it on the client?
Client checks can be bypassed; only the server can be trusted
It makes the page load faster
Server validation is optional and just nice-to-have
✅ Never trust the client. Anyone can bypass front-end checks with curl or dev tools. Server validation is the real defence.

Key takeaways

  • A REST API exposes resources (URLs, named with plural nouns) acted on by HTTP methods.
  • Every request/response is first line · headers · blank line · body. Headers carry metadata (Content-Type, Accept, Authorization).
  • GET/POST/PUT/PATCH/DELETE map to CRUD. PUT replaces, PATCH partially updates. GET is safe; GET/PUT/DELETE are idempotent; POST is neither.
  • Status codes: 2xx success, 3xx redirect, 4xx client's fault, 5xx server's fault. Know 200/201/204, 400/401/403/404/422/429, 500.
  • Data travels as JSON (JSON.stringify / JSON.parse); use one consistent error shape like { error: { message, code } }.
  • Query params (?status=active&page=2&limit=20) filter, sort, and paginate. Always validate on the server — never trust the client.
  • Auth: authentication = "who are you?" (401), authorization = "what may you do?" (403). Tokens flow in the Authorization: Bearer header. Always use HTTPS.
  • CORS is the browser's same-origin policy; the server opts in via headers. Versioning (/api/v1/) and rate limiting (429) keep APIs stable and safe.
  • Test with Postman, the REST Client, or curl — no front-end required. REST is the default style; GraphQL/gRPC come later.
