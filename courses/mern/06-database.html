🗄️ Database: MSSQL

Servers forget everything when they restart. A database stores your data permanently and lets you query it fast. You'll learn SQL — the language of data — and connect Microsoft SQL Server to your Node API.

Why do we need a database?

So far our task list lived in a JavaScript array on the server. The problem: restart the server and it's gone. Real apps need data that persists (survives restarts and crashes), can be searched quickly even with millions of rows, handles many users at once without corrupting each other's work, and stays consistent (no half-finished updates). Doing all of that by hand with files would be a nightmare — a database gives it to you for free.

MSSQL (Microsoft SQL Server) is a relational database: data lives in tables (like spreadsheets) that can be linked together. You talk to it using SQL (Structured Query Language) — a small, English-like language that has barely changed in 40 years. Learn it once and it transfers to PostgreSQL, MySQL, SQLite, and more.

Why "relational"? The big idea

The word relational means tables can relate to one another. Instead of repeating the same facts over and over, you store each fact once and point to it. Imagine storing tasks in a single flat spreadsheet where every row also repeats the owner's full name, email, and team:

The "everything in one table" trap
IdTitleOwnerNameOwnerEmail
1Learn SQLAisha Khanaisha@x.com
2Build APIAisha Khanaisha@x.com
3Ship appAisha Kahn ⚠️aisha@x.com

Aisha's name is duplicated three times — and row 3 has a typo, so now the database "thinks" there are two different people. If she changes her email, you must hunt down every row. This is called an update anomaly, and it's exactly what relational design prevents: store Aisha once in a Users table, and have each task simply reference her Id.

🔑

The golden rule of database design: each fact should live in exactly one place. If you ever find yourself copy-pasting the same value into many rows, that's a hint a second table is hiding in there.

A gentle intro to normalization (1NF · 2NF · 3NF)

Normalization sounds scary but it's just a set of common-sense guidelines for splitting data into well-organised tables. You don't need to memorise them — just absorb the spirit. Here they are in plain English:

RuleIn plain EnglishExample fix
1NF — First Normal FormOne value per cell. No comma-separated lists stuffed into a single column.Instead of Tags = "urgent,home" in one cell, give each tag its own row.
2NF — Second Normal FormEvery column depends on the whole primary key, not just part of it.Don't store a product's name in an order-line table; the name depends on the product, not the order.
3NF — Third Normal FormColumns depend only on the key — not on other non-key columns.Don't store OwnerEmail in Tasks; it depends on the owner, not the task. Put it in Users.

Most well-designed apps aim for 3NF: "every column describes the key, the whole key, and nothing but the key." Get to roughly here and your data will be clean and easy to change. We'll see the practical version of all this when we build relationships and JOINs later.

💡

Normalization keeps data clean, but JOINs cost a little speed. Real systems sometimes denormalize (deliberately duplicate) for performance. As a beginner: normalize first, optimise later only if you measure a real problem.

① Tables, rows & columns

A relational database is organised like a set of spreadsheets. Four words cover almost everything:

  • A table holds one kind of thing (e.g. all Tasks). Think of it as one spreadsheet tab.
  • A column (also called a field or attribute) is a property every item has (e.g. Title, IsDone). Each column has a fixed data type — it can only ever hold that kind of value.
  • A row (also called a record) is one item — one task. Every row has a value for every column.
  • A primary key (usually Id) is the column whose value uniquely identifies each row, like a serial number. No two rows share it.

Here is a real Tasks table. Notice the Id column is highlighted because it's the primary key:

The whole table is sometimes called a schema when we talk about its shape (which columns exist and their types), separate from the data (the rows inside it). Designing good schemas is most of the skill in databases — and that's what the next few sections build up.

📛

Naming tip: pick a convention and stick to it. This course uses PascalCase table names (Tasks, Users) and singular column names. Plural table / singular id (UserId) is a common, readable choice. Consistency matters more than which style you pick.

② Data types in depth Basic

Every column must declare what kind of value it stores. Choosing well keeps your data correct (you can't accidentally put text in a number column) and your storage small. Here are the types you'll actually use, grouped by purpose:

TypeHoldsWhen to use it
INTWhole numbers, about ±2.1 billionDefault for ids, counts, ages. The everyday integer.
BIGINTHuge whole numbers (±9 quintillion)Ids in tables that may exceed 2 billion rows; large counters.
NVARCHAR(n)Variable-length Unicode textYour default for text. The N = supports any language/emoji. Use a real length like NVARCHAR(200), or NVARCHAR(MAX) for long blobs.
VARCHAR(n)Variable text, non-Unicode (1 byte/char)Only for data you know is plain ASCII (some codes). Saves space but can't store é, 日本語, or 🎉.
CHAR(n)Fixed-length text, always n charsFixed codes like a 2-letter country code CHAR(2). Pads shorter values with spaces.
BIT0 or 1 (and NULL)Booleans / flags: IsDone, IsActive.
DATEA date only (no time)Birthdays, due dates — when the clock time doesn't matter.
TIMEA time of day onlyOpening hours, alarm times.
DATETIME2Date and time, high precisionPreferred for timestamps (CreatedAt). Use it instead of the older DATETIME.
DECIMAL(p,s)Exact decimals: p total digits, s after the pointMoney & anything that must be exact. DECIMAL(10,2) = up to 99,999,999.99.
MONEYA currency-flavoured numberExists, but most teams prefer DECIMAL for clarity and portability.

💸

Money gotcha: never store money as FLOAT. Floats are approximate, so 0.1 + 0.2 isn't exactly 0.3 — and a one-cent error in finance is a bug. Always use DECIMAL(p,s).

🌍

Text gotcha: prefer NVARCHAR over VARCHAR unless you have a specific reason. The "N" makes it Unicode, so names, accents, and emoji all just work. Bugs from non-Unicode columns mangling characters are common and annoying.

③ Installing MSSQL (Windows)

You only need to do this once. The goal: a running SQL Server you can connect to, plus a tool to type SQL into. Don't worry if a step looks unfamiliar — just follow along.

  1. SQL Server Express. Download the free Express edition from Microsoft's SQL Server page. During setup, choose the "Basic" installation. Note the server name it gives you (often localhost\SQLEXPRESS).
  2. SSMS (SQL Server Management Studio). A free graphical tool to browse your database and run SQL. Install it separately, then connect to your server name.
  3. Enable SQL login (for Node). So your app can connect with a username/password, enable "SQL Server Authentication" and create a login. Many tutorials use the sa account plus a strong password for local dev.
  4. Create a database. In SSMS, right-click Databases → New Database, name it devlaunch, and you're ready to create tables.
🍎

On Mac/Linux? SQL Server runs great in Docker: docker run -e "ACCEPT_EULA=Y" -e "SA_PASSWORD=Your_strong_Pass1" -p 1433:1433 mcr.microsoft.com/mssql/server. Then connect with the free Azure Data Studio tool.

④ Creating a table with SQL

You design a table once with CREATE TABLE. You list each column, its data type, and any rules it must follow. SQL keywords are traditionally written in UPPERCASE — it's a convention for readability, not a requirement. This creates our Tasks table:

create-table.sql
CREATE TABLE Tasks (
  Id        INT IDENTITY(1,1) PRIMARY KEY,  -- auto-numbered id
  Title     NVARCHAR(200) NOT NULL,         -- required text
  IsDone    BIT NOT NULL DEFAULT 0,         -- defaults to "false"
  CreatedAt DATETIME2 NOT NULL DEFAULT SYSDATETIME()
);
ClauseMeaning
IDENTITY(1,1)Auto-increment: each new row gets the next number automatically
PRIMARY KEYThis column uniquely identifies each row
NOT NULLThis field is required — can't be empty
DEFAULT 0If no value is given, use this
SYSDATETIME()A built-in function returning the current date & time

Once this runs, the table's shape is locked in: every row will have exactly these four columns. The next two sections dig into the rules you can attach to columns (constraints) and how to change a table after it already exists.

⑤ Constraints — rules that protect your data Intermediate

A constraint is a rule the database enforces for you, automatically, on every insert and update. This is huge: instead of hoping your app code always validates correctly, the database guarantees the rule can never be broken — even if a bug, a different app, or a person typing SQL by hand tries to. Constraints are your data's last line of defence.

ConstraintGuaranteesExample
PRIMARY KEYEach row is uniquely identifiable; the column is unique and not null. One per table.Id INT PRIMARY KEY
FOREIGN KEYA value must match an existing row in another table — you can't point at a user who doesn't exist.FOREIGN KEY (UserId) REFERENCES Users(Id)
NOT NULLA value is required; the cell can never be empty.Title NVARCHAR(200) NOT NULL
UNIQUENo two rows can share this value (but it may be null once).Email NVARCHAR(255) UNIQUE
DEFAULTSupplies a value when none is given on insert.IsDone BIT DEFAULT 0
CHECKA custom condition every value must satisfy.CHECK (Priority BETWEEN 1 AND 5)

Here's a richer table that uses several constraints together. Read the comments — each line is a promise the database will keep forever:

users-with-constraints.sql
CREATE TABLE Users (
  Id        INT IDENTITY(1,1) PRIMARY KEY,        -- unique & required, auto-numbered
  Email     NVARCHAR(255) NOT NULL UNIQUE,        -- required, no duplicates
  Name      NVARCHAR(120) NOT NULL,               -- required
  Age       INT NULL CHECK (Age >= 0),            -- optional, but never negative
  Role      NVARCHAR(20) NOT NULL DEFAULT 'member',
  CreatedAt DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
  CHECK (Role IN ('member', 'admin'))             -- only these two values allowed
);

Now if anyone tries to insert a user with a duplicate email, a negative age, or a role of 'wizard', the database rejects it with an error. Your data stays valid no matter what.

🔑

The FOREIGN KEY is the heart of "relational". It says "this column must point at a real row over there." It blocks orphaned data (a task whose owner was deleted) and is what makes JOINs trustworthy. We'll use it heavily in the Relationships section.

⑥ Changing a table with ALTER TABLE

Real apps evolve — you'll need to add a column to an existing table without throwing away the data. ALTER TABLE modifies a table's shape in place:

alter.sql
-- Add a new column (existing rows get NULL or the default)
ALTER TABLE Tasks ADD Priority INT NOT NULL DEFAULT 3;

-- Add a foreign-key column linking tasks to users
ALTER TABLE Tasks ADD UserId INT NULL;
ALTER TABLE Tasks
  ADD CONSTRAINT FK_Tasks_Users
  FOREIGN KEY (UserId) REFERENCES Users(Id);

-- Add a CHECK constraint to an existing table
ALTER TABLE Tasks
  ADD CONSTRAINT CK_Priority CHECK (Priority BETWEEN 1 AND 5);

-- Remove a column you no longer need
ALTER TABLE Tasks DROP COLUMN Priority;

-- Remove a named constraint
ALTER TABLE Tasks DROP CONSTRAINT CK_Priority;
💡

Name your constraints (e.g. FK_Tasks_Users) when you add them. Auto-generated names are random and ugly, which makes them painful to drop later. A clear name pays off the day you need to change it.

🚨

Adding a NOT NULL column to a table that already has rows will fail unless you give it a DEFAULT — otherwise the existing rows would have no value, which the rule forbids. The default fills them in.

⑦ The four SQL commands = CRUD

Just like REST has four methods, SQL has four core commands — and they line up perfectly. If you remember CRUD from the REST module, you already know the shape of SQL:

SQLDoesCRUDREST method
SELECTRead rowsReadGET
INSERTAdd a rowCreatePOST
UPDATEChange rowsUpdatePUT
DELETERemove rowsDeleteDELETE
crud.sql
-- READ: get all tasks
SELECT * FROM Tasks;

-- READ with a filter & sort
SELECT Title, IsDone
FROM Tasks
WHERE IsDone = 0
ORDER BY CreatedAt DESC;

-- CREATE: add a task
INSERT INTO Tasks (Title) VALUES ('Learn JOINs');

-- UPDATE: mark task 2 as done
UPDATE Tasks SET IsDone = 1 WHERE Id = 2;

-- DELETE: remove task 3
DELETE FROM Tasks WHERE Id = 3;
🚨

Never forget the WHERE clause on UPDATE or DELETE. DELETE FROM Tasks; with no WHERE deletes every row. This is the most famous beginner (and senior!) mistake. A safe habit: write the WHERE first, then go back and type DELETE / UPDATE. Many people even run a SELECT with the same WHERE to preview which rows will be hit before changing them.

🆔

INSERT doesn't set Id — the IDENTITY(1,1) column auto-numbers it. To get the id of the row you just inserted, add OUTPUT INSERTED.Id to the statement (you'll see this used from Node later).

⑧ SELECT in depth — the workhorse Intermediate

You'll write SELECT far more than any other command — reading data is most of what apps do. Let's unpack every part of it, one small idea at a time. All examples run against our familiar Tasks table.

Choosing & renaming columns: aliases with AS

List the columns you want instead of *, and rename them in the output with AS. (Selecting only what you need is also faster.)

aliases.sql
SELECT Title AS TaskName, IsDone AS Completed
FROM Tasks;
-- Output columns are now labelled "TaskName" and "Completed"

DISTINCT — drop duplicate rows

distinct.sql
-- List each distinct completion status (returns just 0 and 1)
SELECT DISTINCT IsDone FROM Tasks;

TOP (N) — limit how many rows come back

top.sql
-- The 3 most recently created tasks
SELECT TOP (3) * FROM Tasks ORDER BY CreatedAt DESC;
📄

TOP is SQL Server's way to limit rows (other databases use LIMIT). Combined with ORDER BY ... OFFSET ... FETCH, it powers "page 2 of results" in real apps.

Filtering with WHERE — all the operators

The WHERE clause keeps only the rows that match a condition. Here is the full toolkit:

OperatorMeaningExample
=EqualsWHERE IsDone = 1
<>Not equal toWHERE IsDone <> 1
> < >= <=Greater / less than (or equal)WHERE Id >= 2
BETWEEN a AND bIn a range, inclusiveWHERE Id BETWEEN 2 AND 4
IN (...)Matches any value in a listWHERE Id IN (1, 3, 5)
LIKEText pattern matchWHERE Title LIKE 'Learn%'
IS NULLHas no valueWHERE UserId IS NULL

For LIKE, the wildcards are % (any number of characters, including none) and _ (exactly one character):

like.sql
SELECT * FROM Tasks WHERE Title LIKE 'Learn%';   -- starts with "Learn"
SELECT * FROM Tasks WHERE Title LIKE '%app%';    -- contains "app"
SELECT * FROM Tasks WHERE Title LIKE '%SQL';     -- ends with "SQL"

Combining conditions: AND, OR, NOT

and-or.sql
-- Unfinished tasks created on or after a date
SELECT * FROM Tasks
WHERE IsDone = 0 AND CreatedAt >= '2026-06-19';

-- Task is either id 1, OR is already done
SELECT * FROM Tasks
WHERE Id = 1 OR IsDone = 1;

-- Use parentheses to control grouping, just like maths
SELECT * FROM Tasks
WHERE (IsDone = 0 OR Id = 1) AND Title LIKE '%a%';
⚠️

AND binds tighter than OR. A OR B AND C means A OR (B AND C) — probably not what you meant. When mixing them, always add parentheses so your intent is unmistakable.

Sorting with ORDER BY

order-by.sql
SELECT * FROM Tasks ORDER BY Title ASC;            -- A→Z (ASC is the default)
SELECT * FROM Tasks ORDER BY CreatedAt DESC;       -- newest first
SELECT * FROM Tasks ORDER BY IsDone, CreatedAt DESC; -- group by done, then newest
🧪

Several of these run in the live playground just below (anything that's a single-table SELECT on Tasks with WHERE using = < > >= <= <> and ORDER BY). Patterns like LIKE, IN, and BETWEEN aren't supported by the tiny demo engine — run those in SSMS.

⑨ Working with NULL — the "unknown" value Intermediate

NULL is special and trips up almost every beginner, so let's be crystal clear. NULL means "no value / unknown" — it is not zero, and not an empty string ''. It's the database saying "I don't know what goes here."

Because NULL is "unknown," you can't compare it with =. Ask "is this column equal to NULL?" and the answer is itself unknown — never true. So WHERE UserId = NULL returns nothing, even for rows that are null! You must use the special IS NULL / IS NOT NULL tests:

null-checks.sql
SELECT * FROM Tasks WHERE UserId IS NULL;      -- unassigned tasks ✅
SELECT * FROM Tasks WHERE UserId IS NOT NULL;  -- assigned tasks ✅
SELECT * FROM Tasks WHERE UserId = NULL;       -- ❌ ALWAYS returns 0 rows!

Supplying a fallback: ISNULL & COALESCE

Often you want "use the real value, but if it's NULL show this instead." Two functions do that:

coalesce.sql
-- If UserId is NULL, show 0 instead
SELECT Title, ISNULL(UserId, 0) AS Owner FROM Tasks;

-- COALESCE returns the first non-NULL of several options
SELECT COALESCE(Nickname, FullName, 'Anonymous') AS DisplayName
FROM Users;

ISNULL(a, b) takes exactly two arguments; COALESCE(a, b, c, ...) takes many and is standard SQL, so it's the more flexible choice.

🧠

Three-valued logic gotcha: SQL conditions can be true, false, or unknown. A NULL involved in a comparison makes the result "unknown," and WHERE only keeps rows that are true. So a row can be silently excluded by both WHERE x = 5 and WHERE x <> 5 when x is NULL. When NULLs are possible, handle them explicitly.

⑪ Aggregates & GROUP BY — summarising data Intermediate

So far we've read individual rows. Often you instead want a summary: "how many tasks?", "what's the total?", "the average price per category?". Aggregate functions crunch many rows into a single answer:

FunctionAnswers
COUNT(*)How many rows?
SUM(col)Total of a numeric column
AVG(col)Average value
MIN(col) / MAX(col)Smallest / largest value
aggregate.sql — run in SSMS
SELECT COUNT(*) AS TotalTasks FROM Tasks;            -- e.g. 4
SELECT COUNT(*) AS Done FROM Tasks WHERE IsDone = 1; -- e.g. 2
SELECT MIN(CreatedAt) AS Earliest FROM Tasks;        -- oldest task date

GROUP BY — one summary per group

The real power comes when you split rows into groups and summarise each one. "How many tasks are done vs not done?" groups by IsDone, then counts within each group:

group-by.sql — run in SSMS
SELECT IsDone, COUNT(*) AS HowMany
FROM Tasks
GROUP BY IsDone;

That query returns one row per distinct IsDone value:

12

HAVING vs WHERE

This is a classic exam question. WHERE filters individual rows before grouping; HAVING filters the groups after aggregating. Use HAVING when your condition involves an aggregate like COUNT():

having.sql — run in SSMS
-- "Which owners have more than 2 tasks?"
SELECT UserId, COUNT(*) AS TaskCount
FROM Tasks
WHERE IsDone = 0          -- 1) keep only unfinished rows (per-row filter)
GROUP BY UserId           -- 2) group what remains by owner
HAVING COUNT(*) > 2;      -- 3) keep only groups with > 2 tasks (per-group filter)
🔑

Order of operations to memorise: WHERE → GROUP BY → HAVING → SELECT → ORDER BY. The database filters rows, groups them, filters groups, picks columns, then sorts. You can't use HAVING without aggregating, and you can't put an aggregate in WHERE.

⑫ Relationships & JOINs Advanced

The "relational" superpower: link tables together. Say each task belongs to a user. We store the user's id in the task as a foreign key, instead of duplicating the user's name everywhere. This is a one-to-many relationship: one user has many tasks, but each task has exactly one user.

A JOIN stitches rows from two tables together by matching the foreign key to the primary key — "give me each task with its owner's name":

join.sql
SELECT Tasks.Title, Users.Name AS Owner
FROM Tasks
INNER JOIN Users ON Tasks.UserId = Users.Id;

-- Result:
-- Title       | Owner
-- ------------+-------
-- Learn SQL   | Aisha
-- Build API   | Ravi
-- Ship app    | Aisha

The ON clause is the matching rule: "line up rows where the task's UserId equals the user's Id." A handy shortcut is to alias the tables with short letters so you type less:

join-alias.sql
SELECT t.Title, u.Name AS Owner
FROM Tasks AS t
INNER JOIN Users AS u ON t.UserId = u.Id;

The four kinds of JOIN

JOINs differ in what they do with rows that have no match on the other side. Picture two overlapping circles (Tasks and Users); each JOIN keeps a different region:

JOIN typeKeepsUse when…
INNER JOINOnly rows that match on both sides.You want pairs only — tasks that have an owner.
LEFT JOINEvery row from the left table; matched data from the right, else NULL.Most common outer join. "All tasks, plus owner name if one exists." Unassigned tasks still appear.
RIGHT JOINEvery row from the right table; left side may be NULL.Rare — usually you just flip the tables and use LEFT.
FULL OUTER JOINEvery row from both; unmatched cells are NULL.Reconciling two lists — show users with no tasks and tasks with no user.

Here's why LEFT JOIN matters. Suppose task "Ship app" has no owner yet (UserId IS NULL). An INNER JOIN would silently drop it; a LEFT JOIN keeps it, with a NULL owner:

Foreign keys + JOINs are how databases model the real world: orders belong to customers, comments belong to posts, tasks belong to users. No data duplication, no inconsistency. Reach for LEFT JOIN by default when you don't want to accidentally hide rows that lack a match.

⑬ Many-to-many: the junction table Advanced

One-to-many is easy: put the foreign key on the "many" side. But what about many-to-many? A task can have many tags, and a tag can apply to many tasks. There's no single column that can hold "many" ids — so we introduce a third table that sits in the middle, called a junction (or bridge / link) table.

junction.sql
CREATE TABLE Tags (
  Id   INT IDENTITY(1,1) PRIMARY KEY,
  Name NVARCHAR(50) NOT NULL UNIQUE
);

-- The junction table: each row is one (task, tag) pairing
CREATE TABLE TaskTags (
  TaskId INT NOT NULL FOREIGN KEY REFERENCES Tasks(Id),
  TagId  INT NOT NULL FOREIGN KEY REFERENCES Tags(Id),
  PRIMARY KEY (TaskId, TagId)     -- composite key: no duplicate pairings
);

To list each task with its tags, JOIN across all three tables:

tags-join.sql — run in SSMS
SELECT t.Title, tag.Name AS Tag
FROM Tasks AS t
JOIN TaskTags AS tt ON tt.TaskId = t.Id
JOIN Tags     AS tag ON tag.Id    = tt.TagId;
🔑

Whenever you hear "each X can have many Y, and each Y can have many X," reach for a junction table. Students↔Courses, Posts↔Tags, Users↔Roles — all the same pattern.

⑭ Subqueries — a query inside a query Advanced

Sometimes the value you want to filter by is itself the result of a query. A subquery is a SELECT nested inside another statement, usually in parentheses. The inner query runs first; its result feeds the outer one.

subquery.sql — run in SSMS
-- "Tasks owned by users named Aisha"
SELECT * FROM Tasks
WHERE UserId IN (SELECT Id FROM Users WHERE Name = 'Aisha');

-- The inner query finds Aisha's id(s); the outer keeps tasks with those UserIds.

-- A subquery can also compute a single value:
SELECT * FROM Tasks
WHERE CreatedAt = (SELECT MAX(CreatedAt) FROM Tasks);  -- the newest task
💡

Many subqueries can be rewritten as JOINs, and JOINs are often faster. But subqueries can be clearer to read for "filter by a list" style questions. Learn both; reach for whichever makes the query easier to understand.

⑮ Indexes — making reads fast Advanced

Imagine finding a word in a 900-page textbook. Without an index you'd read every page (a full table scan). With the index at the back, you jump straight to the right page. A database index works exactly the same way: it's a sorted lookup structure that lets the engine find rows without scanning the whole table.

You create an index on the columns you frequently search or join on:

index.sql
-- Speeds up: SELECT ... WHERE UserId = ...  and JOINs on UserId
CREATE INDEX IX_Tasks_UserId ON Tasks(UserId);
🔑

Your primary key automatically gets an index — that's why looking up a row by Id is always fast. You add extra indexes for the other columns you filter or join on a lot.

⚖️

The trade-off: indexes speed up reads but slow down writes — every INSERT/UPDATE/DELETE must also keep each index up to date, and indexes take disk space. So index the columns you search on, not every column. Index thoughtfully, not greedily.

⑯ Transactions & ACID Advanced

Some operations involve several steps that must all succeed or all fail together — never halfway. The classic example is a bank transfer: subtract money from one account, add it to another. If the power dies between those two steps, money would vanish. A transaction wraps the steps so they're treated as one indivisible unit.

transfer.sql
BEGIN TRANSACTION;

UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1;
UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2;

-- If both worked, lock it in:
COMMIT;

-- If something went wrong, undo everything instead:
-- ROLLBACK;

Transactions give you the ACID guarantees — four promises, in plain English:

LetterPromise
AtomicityAll steps happen, or none do. No half-finished updates.
ConsistencyThe database moves from one valid state to another — constraints are never left broken.
IsolationConcurrent transactions don't see each other's half-done work; it's as if they ran one at a time.
DurabilityOnce committed, the change survives crashes and power loss — it's written to disk for good.
🏦

Use a transaction whenever two or more changes must agree: transferring funds, placing an order while reducing stock, or creating a user and their default settings together. Either both happen or neither does.

⑰ Views & stored procedures Advanced

Views — a saved query

A view is a query you give a name to, then use like a table. It stores no data of its own — it's a saved SELECT that runs fresh each time. Views hide complexity: write the gnarly JOIN once, then everyone just selects from the simple-looking view.

view.sql
CREATE VIEW TaskOwners AS
SELECT t.Id, t.Title, u.Name AS Owner
FROM Tasks AS t
LEFT JOIN Users AS u ON t.UserId = u.Id;

-- Now use it like any table:
SELECT * FROM TaskOwners WHERE Owner = 'Aisha';

Stored procedures — saved logic

A stored procedure is a named block of SQL (which can take parameters and contain logic) stored inside the database. You call it by name. Teams use them to centralise important logic, reduce repeated SQL, and sometimes for performance.

procedure.sql
CREATE PROCEDURE AddTask
  @Title NVARCHAR(200)
AS
BEGIN
  INSERT INTO Tasks (Title) VALUES (@Title);
END;

-- Call it:
EXEC AddTask @Title = 'Write tests';
💬

You don't need procedures for this course — plain parameterised queries from Node are perfectly fine and very common. But you'll meet procedures on real teams, so it's good to recognise them.

⑱ Querying MSSQL from Node

The mssql npm package connects Node to SQL Server. Install it with npm install mssql, then create a shared connection pool and run queries.

Why a pool? Opening a fresh database connection is slow (a handshake every time). A pool opens a handful of connections once, keeps them warm, and hands them out as requests come in. You create it a single time for the whole app and reuse it everywhere — never open a new connection per request.

db/pool.js
import sql from "mssql";

const config = {
  server: "localhost",
  database: "devlaunch",
  user: "sa",
  password: process.env.DB_PASSWORD,   // from your .env file
  options: { instanceName: "SQLEXPRESS", encrypt: true, trustServerCertificate: true },
};

// One shared pool of connections for the whole app
export const poolPromise = sql.connect(config);
routes/tasks.js — now backed by MSSQL
import express from "express";
import sql from "mssql";
import { poolPromise } from "../db/pool.js";

const router = express.Router();

// READ all
router.get("/", async (req, res) => {
  const pool = await poolPromise;
  const result = await pool.request().query("SELECT * FROM Tasks ORDER BY Id");
  res.json(result.recordset);     // recordset = the rows
});

// CREATE — note the SAFE, parameterised input
router.post("/", async (req, res) => {
  const { title } = req.body;
  const pool = await poolPromise;
  const result = await pool.request()
    .input("title", sql.NVarChar, title)              // bind, don't concatenate!
    .query("INSERT INTO Tasks (Title) OUTPUT INSERTED.* VALUES (@title)");
  res.status(201).json(result.recordset[0]);
});

export default router;

Reading the result: result.recordset

Every query returns a result object. The rows live in result.recordset — an ordinary JavaScript array of plain objects, so you can use .map(), .length, destructuring, and everything else you already know:

recordset.js
const result = await pool.request()
  .input("id", sql.Int, 2)
  .query("SELECT * FROM Tasks WHERE Id = @id");

result.recordset;        // → [ { Id: 2, Title: "Build the API", ... } ]
result.recordset[0];     // → the first (and only) row
result.recordset.length; // → 1
result.rowsAffected;     // → [1]  (handy for UPDATE/DELETE)

Handling errors

Anything can fail — the DB might be down, a constraint might reject the data. Wrap queries in try/catch and return a sensible HTTP status instead of crashing the server:

error-handling.js
router.get("/:id", async (req, res) => {
  try {
    const pool = await poolPromise;
    const result = await pool.request()
      .input("id", sql.Int, Number(req.params.id))
      .query("SELECT * FROM Tasks WHERE Id = @id");

    if (result.recordset.length === 0) {
      return res.status(404).json({ error: "Task not found" });
    }
    res.json(result.recordset[0]);
  } catch (err) {
    console.error(err);                       // log the real error server-side
    res.status(500).json({ error: "Database error" }); // generic message to client
  }
});

Running a transaction from Node

For multi-step operations that must succeed together, the mssql package exposes a transaction object. Begin it, run each request on that transaction, then commit — or roll back if anything throws:

transaction.js
const pool = await poolPromise;
const tx = new sql.Transaction(pool);
await tx.begin();
try {
  await tx.request()
    .input("id", sql.Int, 1)
    .query("UPDATE Accounts SET Balance = Balance - 100 WHERE Id = @id");
  await tx.request()
    .input("id", sql.Int, 2)
    .query("UPDATE Accounts SET Balance = Balance + 100 WHERE Id = @id");
  await tx.commit();          // both succeeded — make it permanent
} catch (err) {
  await tx.rollback();        // something failed — undo everything
  throw err;
}
🔗

This is the final link in the chain. React → fetch → Express route → pool.request().query() → MSSQL → rows → JSON → back to React. Every module you've done connects right here.

⑲ Critical: SQL injection & how to avoid it Advanced

The single most important security lesson in this course. Never build SQL by gluing user input into a string. A malicious user could inject their own SQL.

❌ Dangerous

// NEVER do this:
query(
  "SELECT * FROM Users WHERE name = '"
  + userInput + "'"
);
// userInput = "'; DROP TABLE Users;--"
// → your table is gone 💀

✅ Safe (parameterised)

// ALWAYS do this:
pool.request()
  .input("name", sql.NVarChar, userInput)
  .query(
    "SELECT * FROM Users WHERE name = @name"
  );
// the driver safely escapes the input
🛡️

Parameterised queries (using .input() and @placeholders) keep user data and SQL commands separate, so input can never become code. Make this a habit from day one — it's not optional.

🧠

Why it works: with a parameter, the database receives the SQL and the value down two separate channels. The query plan is fixed before your value arrives, so the value is always treated as plain data — never as commands to run. There is no string to "break out of." This is the gold standard, used by every serious app.

⑳ ORMs, query builders & data-modeling tips Intermediate

You now know raw SQL, which is the foundation everything else is built on. In the wild you'll also meet tools that write SQL for you, so it helps to know what they are and when to reach for them.

ToolWhat it isFeel
PrismaA modern, type-safe ORM (Object-Relational Mapper). You define models; it generates a typed client.prisma.task.findMany()
SequelizeA long-established ORM for Node with models and associations.Task.findAll()
KnexA query builder — closer to SQL, you chain methods that map almost 1:1 to SQL clauses.knex('Tasks').where('IsDone', 0)

Raw SQL vs a tool: raw SQL gives you full control and zero magic — great for learning and for complex queries. ORMs save boilerplate, add type-safety, and handle migrations, which shines on big apps with many tables. A common, healthy approach: use an ORM/builder for everyday CRUD, and drop to raw SQL for the few gnarly reporting queries. There's no wrong choice — just understand what's running underneath, which you now do.

🧭

Even with an ORM, parameterisation still protects you — the tool parameterises automatically. The danger returns the moment you hand-build a raw SQL string, so the lesson from the last section never expires.

Practical data-modeling tips

  • Give every table a primary key — usually an IDENTITY Id. It makes rows addressable and JOINs reliable.
  • Each fact in one place. If you're copy-pasting a value across rows, a second table probably wants to exist (remember normalization).
  • Use foreign keys to express relationships — let the database guard integrity, don't rely on app code alone.
  • Name consistently. Pick a convention (e.g. PascalCase tables, UserId for foreign keys) and never deviate. Future-you will be grateful.
  • Pick the right type — NVARCHAR for text, DECIMAL for money, DATETIME2 for timestamps, BIT for flags.
  • Add CreatedAt / UpdatedAt timestamps to most tables — they're invaluable for debugging and sorting.
  • Index what you search, not everything. Measure before optimising.

Key takeaways

  • A database stores data permanently; MSSQL is a relational DB of tables (rows & columns). Relational design keeps each fact in one place — the spirit of normalization.
  • Pick the right data type per column (NVARCHAR for text, DECIMAL for money, DATETIME2 for timestamps, BIT for flags).
  • Constraints (PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, DEFAULT, CHECK) let the database guarantee your data stays valid.
  • SQL commands — SELECT / INSERT / UPDATE / DELETE — are CRUD, mapping to REST's GET/POST/PUT/DELETE.
  • SELECT goes deep: aliases, DISTINCT, TOP, rich WHERE operators (BETWEEN, IN, LIKE, IS NULL), and ORDER BY. Remember NULL means "unknown" — test it with IS NULL.
  • Aggregates (COUNT/SUM/AVG/MIN/MAX) with GROUP BY summarise data; HAVING filters groups, WHERE filters rows.
  • A primary key identifies each row; a foreign key + JOIN links tables. Use LEFT JOIN to keep unmatched rows; use a junction table for many-to-many.
  • Indexes speed reads (at a write cost); transactions give all-or-nothing ACID safety.
  • Connect from Node with the mssql package and a shared connection pool; read rows from result.recordset and wrap queries in try/catch.
  • Always use parameterised queries (.input()) to prevent SQL injection — and always include WHERE on UPDATE/DELETE.
