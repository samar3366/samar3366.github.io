🟩 Node.js: JavaScript on the Server

Browsers run JavaScript for the front-end. Node.js lets the same language run on a server — so you can build the back-end with skills you already have. This module is hands-on in your terminal.

🖥️

This module runs on your machine, not in the browser. Code along in VS Code with a terminal open. If you haven't installed Node yet, see Foundations → Toolkit.

What is Node.js, really?

Node.js is a runtime — a program that can execute JavaScript outside a browser. The browser gives JS access to the page (the DOM); Node gives JS access to the computer: files, the network, databases, and the ability to act as a web server.

🔑

Because front-end and back-end are both JavaScript, you don't switch languages, mental models, or tools. This is the core advantage of the "MERN-style" stack you're learning.

Under the hood: the same engine your browser uses

Here's a fun fact that makes Node click for a lot of people: Node.js is built on V8 — the very same JavaScript engine inside Google Chrome. Someone took that engine out of the browser, wrapped it in a layer that can touch files and networks, and shipped it as a standalone program called node. So when you run node server.js, you're running your JavaScript through the exact compiler that powers a browser tab — just pointed at your computer instead of a web page.

What Node adds on top of V8 are the things a server needs and a browser deliberately forbids: reading and writing files, opening network sockets, spawning other programs, reading environment variables, and so on. And what Node removes are the browser-only globals: there is no window, no document, no localStorage, no alert. If you try to use them in Node you'll get a ReferenceError — a common surprise for people coming straight from front-end work.

🌐 Browser globals

window, document, fetch, localStorage, alert. Sandboxed for safety — a web page must not read your hard drive.

🖥️ Node globals

process, __dirname, Buffer, global, plus core modules like fs and http. Full machine access — it's your server.

📌

Node is not a framework or a language. It's a runtime — an environment that executes JavaScript. Express, on the other hand, is a framework (a library you install) that runs inside Node. Keeping that distinction straight prevents a lot of early confusion.

The event loop & non-blocking I/O Intermediate

This is the single most important idea for understanding why Node is so good at building servers. Don't worry — it sounds academic, but the core of it is a simple, everyday idea. Let's build it up with an analogy.

Imagine a coffee shop with one barista. A blocking barista takes your order, then stands frozen at the espresso machine until your drink is 100% done before even looking at the next customer. A non-blocking barista takes your order, starts the machine, and immediately turns to the next customer while the first drink brews — circling back the moment a drink is ready. Same one barista, but the second shop serves a huge line without anyone standing idle.

Node is that second barista. It runs your JavaScript on a single main thread (one barista), but whenever it hits a slow task — reading a file, querying a database, waiting on the network — it doesn't stand and wait. It hands that slow task off to the system, registers a "call me back when it's done" note, and immediately moves on to other work. The mechanism that keeps checking "is anything finished yet?" and runs the matching callback is called the event loop.

This is why people say Node is great for I/O-heavy work: APIs, web servers, chat apps, anything that spends most of its time waiting on databases or networks rather than crunching numbers. While one request waits on the database, Node happily serves dozens of others. That's the secret behind handling many connections with very little memory.

⚠️

The flip side — never block the loop. Because there's only one main thread, a single long synchronous task (a giant for loop, JSON.parse on a 50 MB string, or the synchronous fs.readFileSync in a busy server) freezes every user at once — the frozen barista. Prefer the async versions (await readFile(...)), and keep heavy CPU work out of the request path.

How to read it: the Call Stack is the single barista — only what's here is actually executing. Slow tasks go to the Background; when they finish, their callbacks line up in a queue. ① Running your first Node program

Create a file hello.js and run it with node hello.js. Note: in Node there's no alert or document — you print with console.log, which writes to the terminal.

hello.js
const name = "World";
console.log(`Hello, ${name}! Running on Node ${process.version}`);

// process.argv holds command-line arguments
console.log("You ran:", process.argv[1]);
terminal
node hello.js
// → Hello, World! Running on Node v20.x.x
💡

All the JavaScript you learned in Module 1 — variables, functions, arrays, map, async/await — works identically here. You're not learning a new language, just a new environment.

Two ways to run JavaScript: the script and the REPL

There are two everyday ways to execute code with Node, and you'll use both:

  1. Run a file. node hello.js executes the whole file top to bottom and exits. This is how you run real programs and servers.
  2. The REPL (interactive shell). Type just node with no filename and you get a live prompt where you can type one expression at a time and see the result instantly — perfect for trying out a quick idea, like a browser console but in your terminal. Type .exit or press Ctrl+C twice to leave.
🧪

REPL stands for Read–Eval–Print Loop: it reads what you type, evaluates it, prints the answer, and loops back for more. Great for sanity-checking "does [1,2,3].includes(2) return true?" without creating a file.

Passing arguments & process.argv

Programs often need input. The process object (a Node global, no import needed) exposes process.argv — an array of the words typed on the command line. The first two entries are always the path to node and the path to your script; your arguments start at index 2.

greet.js
// Run as:  node greet.js Alice
const args = process.argv.slice(2);   // drop node + script path
const who = args[0] || "stranger";
console.log(`Hello, ${who}!`);        // → Hello, Alice!

② Modules — splitting code across files

Real apps span many files. Modules let one file export things and another import them. Think of it like a kitchen: instead of one enormous 5,000-line recipe, you keep ingredients in labelled jars (modules) and grab exactly what you need. Each file is its own private scope — variables don't leak between files unless you explicitly export them, which keeps things tidy and prevents name clashes.

Node actually supports two module systems, and you'll bump into both in the wild. Knowing the difference saves you from a category of baffling errors.

📦 CommonJS (CJS) — the original

Uses require() and module.exports. The default for older Node code and most npm tutorials. Files often end in .js (or .cjs).

✨ ES Modules (ESM) — the modern standard

Uses import / export — the same syntax you used in React. The web standard, and what we use in this course. Enabled with "type":"module".

The same module, both ways

CommonJS
// math.cjs — export
function add(a, b) { return a + b; }
module.exports = { add };

// app.cjs — import
const { add } = require("./math.cjs");
console.log(add(2, 3)); // 5
ES Modules
// math.js — export
export function add(a, b) { return a + b; }

// app.js — import
import { add } from "./math.js";
console.log(add(2, 3)); // 5

Below is the standard ESM form we'll use everywhere from here on. Notice the two flavours of export: named exports (any number per file, imported by exact name in { }) and a single default export (imported under any name you like).

math.js — exports
export function add(a, b) {
  return a + b;
}
export const PI = 3.14159;
app.js — imports
import { add, PI } from "./math.js";       // named imports — exact names
import myMath, { add as plus } from "./bundle.js"; // default + rename

console.log(add(2, 3));   // 5
console.log(PI);          // 3.14159
console.log(plus(4, 4));  // 8  (renamed with "as")
bundle.js — a default export
export function add(a, b) { return a + b; }   // named
// exactly ONE default export allowed per file:
export default { version: "1.0", name: "myMath" };

⚠️

To use import/export, add "type": "module" to your package.json. Two beginner gotchas in ESM: (1) you must include the file extension in relative imports — "./math.js", not "./math"; and (2) there's no require, __dirname, or __filename by default (you recreate the last two from import.meta.url if needed). Mixing the two systems is the source of the dreaded "Cannot use import statement outside a module" error — it almost always means you forgot "type":"module".

🧭

How to choose: for any new project, use ES Modules — it's the standard, matches React, and supports top-level await. You'll only reach for CommonJS when maintaining older code or using a rare package that hasn't modernised yet.

③ The built-in toolbox: core modules Intermediate

Before you install a single package from npm, Node already ships with a generous standard library — the core modules. These are built in: you just import them, no installation needed. Knowing what's already in the box stops you from installing a package for something Node can do natively. Here's a tour of the ones you'll actually reach for.

fs / fs/promises — the file system

Read, write, append, delete, and inspect files and folders. Always prefer the promise-based version (fs/promises) so you can await instead of nesting callbacks — and so you don't block the event loop.

fs-tour.js
import { readFile, writeFile, mkdir, readdir } from "fs/promises";

await mkdir("data", { recursive: true });           // make a folder
await writeFile("data/log.txt", "first line\n");    // create/overwrite
const files = await readdir("data");                // list a folder
console.log(files);                                 // → ["log.txt"]

path — build file paths safely

Never glue paths together with "folder/" + name — Windows uses \ and macOS/Linux use /, and you'll create bugs. The path module joins paths correctly for whatever OS you're on.

path-tour.js
import path from "path";

path.join("routes", "tasks.js");   // "routes/tasks.js" (or routes\tasks.js on Windows)
path.resolve("data", "log.txt");   // absolute path from the current dir
path.basename("/app/server.js");   // "server.js"
path.extname("photo.PNG");         // ".PNG"

os, url, process — the environment

os

Info about the machine: os.platform(), os.cpus().length, os.totalmem(), os.homedir().

url

Parse and build URLs: new URL("https://x.com/a?id=5") gives you .pathname, .searchParams, etc.

The process object (global — no import) is your program's window into the outside world: process.env (environment variables), process.argv (CLI args), process.cwd() (current working directory), and process.exit(code) (stop the program; 0 means success, anything else signals an error).

events — the EventEmitter pattern

A huge amount of Node is built on a simple publish/subscribe idea: one part of your code emits a named event, and other parts listen for it. You already met this style in the browser with button.addEventListener("click", …). Node's EventEmitter is the same concept, and Express's app and the HTTP server are themselves emitters under the hood.

events-tour.js
import { EventEmitter } from "events";

const bus = new EventEmitter();

// listen ("subscribe")
bus.on("order", (item) => {
  console.log(`👨‍🍳 Cooking a ${item}`);
});

// emit ("publish") — triggers every listener
bus.emit("order", "pizza");   // → 👨‍🍳 Cooking a pizza
bus.emit("order", "salad");   // → 👨‍🍳 Cooking a salad
🔑

Listeners run in the order they were registered, and emitting an event with no listeners simply does nothing. This "fire an event, whoever cares responds" pattern keeps parts of a system loosely coupled — the emitter doesn't need to know who's listening.

④ npm — the world's package library

npm (Node Package Manager) installs reusable code packages written by others, so you don't reinvent the wheel. There are over two million of them — for servers, dates, validation, you name it. npm comes bundled with Node, so once you've installed Node you already have it. When someone says "there's a package for that," npm is where it lives.

  1. Start a project. In an empty folder, run npm init -y. This creates package.json, the "ID card" of your project.
  2. Install a package. npm install express downloads Express and records it in package.json. Packages land in a node_modules folder (never edit or commit that folder).
  3. Use it. import express from "express"; and you're off.
package.json (after installing)
{
  "name": "my-server",
  "type": "module",
  "scripts": {
    "dev": "node --watch server.js"
  },
  "dependencies": {
    "express": "^4.19.2"
  }
}

Anatomy of package.json

This little file is the heart of every Node project. Let's read it field by field — once it stops looking like magic, npm makes a lot more sense.

package.json — annotated
{
  "name": "my-server",          // project name (lowercase, no spaces)
  "version": "1.0.0",           // YOUR app's version (semver)
  "type": "module",             // use import/export (ESM)
  "main": "server.js",          // the entry-point file
  "scripts": {                  // named shortcuts, run via "npm run"
    "start": "node server.js",
    "dev": "node --watch server.js"
  },
  "dependencies": {             // needed to RUN the app
    "express": "^4.19.2"
  },
  "devDependencies": {          // only needed while DEVELOPING
    "nodemon": "^3.1.0"
  }
}

📦 dependencies

Packages your app needs to run in production — Express, a database driver, etc. Installed with npm install express.

🛠️ devDependencies

Tools only used while building — test runners, nodemon, linters. The end user never needs them. Installed with npm install -D nodemon (the -D flag).

Semantic versioning: what ^ and ~ mean

A version like 4.19.2 is MAJOR.MINOR.PATCH. The prefix in front of it tells npm how much it's allowed to auto-upgrade when someone runs npm install:

You wrotenpm may installMeaning
^4.19.24.19.2 up to <5.0.0Caret — new minor + patch fixes (the common default)
~4.19.24.19.2 up to <4.20.0Tilde — only patch fixes, stricter
4.19.2exactly 4.19.2Pinned — never changes

The version rules are: bump PATCH for bug fixes, MINOR for new backward-compatible features, and MAJOR for breaking changes. So a caret (^) says "give me fixes and features, but never a breaking major version."

🔒

So what's package-lock.json? Your package.json says "^4.19.2" (a range), but the lock file records the exact version that actually got installed, down to every nested dependency. Commit it! It guarantees your teammate's npm install produces the identical node_modules as yours — no more "works on my machine."

Local vs global, and npx

  • Local install (npm install express): goes into this project's node_modules. The normal case — each project keeps its own copies, so versions can't conflict.
  • Global install (npm install -g some-cli): installed once for your whole machine, usable from any folder. Reserve this for command-line tools, not app dependencies.
  • npx: runs a package without permanently installing it — e.g. npx create-vite@latest. Perfect for one-off scaffolding tools you don't want cluttering your machine.
⚡

node --watch server.js auto-restarts your server when you save a file — like Vite's hot reload, but for the back-end.

🚫

Never commit node_modules to Git. It can be hundreds of megabytes and is fully rebuildable from package.json + package-lock.json. Add it to .gitignore. Anyone who clones your repo just runs npm install to recreate it.

⑤ Async patterns & the file system

Node tasks like reading files take time, so they're async — exactly the async/await you learned in Module 1. But since you'll read other people's Node code constantly, it helps to know the three generations of async style you'll encounter, because they all do the same job in increasingly readable ways.

1. Callbacks (the original) & the "error-first" rule

The oldest Node style passes a function to be called when the work finishes. Node's convention is the error-first callback: the callback's first argument is always an error (or null if all went well), and the result comes second. You check the error before trusting the data.

callback-style.js
import { readFile } from "fs";   // callback version (no /promises)

readFile("note.txt", "utf-8", (err, text) => {
  if (err) {                     // ALWAYS check error first
    console.error("Failed:", err.message);
    return;
  }
  console.log("Got:", text);     // only safe to use after the check
});
😵

Nesting many callbacks creates the infamous "callback hell" — a staircase of code drifting off the right edge of the screen. Promises and async/await were invented to fix exactly this.

2. Promises

A Promise represents a value that isn't ready yet — it will eventually resolve with a result or reject with an error. You chain .then() for success and .catch() for failure, flattening the staircase into a readable pipeline.

promise-style.js
import { readFile } from "fs/promises";

readFile("note.txt", "utf-8")
  .then((text) => console.log("Got:", text))
  .catch((err) => console.error("Failed:", err.message));

3. async / await (what we use)

This is just promises with nicer syntax — it reads like ordinary top-to-bottom code, but pauses at each await until the promise settles. Wrap it in try/catch to handle errors, the same way you'd catch a thrown exception.

files.js
import { readFile, writeFile } from "fs/promises";

async function main() {
  try {
    await writeFile("note.txt", "Hello from Node!");     // write a file
    const text = await readFile("note.txt", "utf-8");     // read it back
    console.log("File says:", text);
  } catch (err) {
    console.error("Something went wrong:", err.message);  // one place catches all awaits
  }
}

main();
💡

This is why Module 1's async section mattered: every back-end operation — files, network, database queries — is asynchronous. await is your everyday tool here. A single try/catch can guard several await calls at once — if any one rejects, control jumps straight to catch.

You can practice the try/catch-around-await shape right here — this is genuine browser-runnable JavaScript using a promise that rejects:

Console
⚠️

Unhandled rejections. If a promise rejects and nobody is there to .catch() it (or no try/catch around the await), Node prints an UnhandledPromiseRejection warning and may crash. The fix is simply to always handle errors — and you can add a final safety net at the top of your app:

safety-net.js
process.on("unhandledRejection", (reason) => {
  console.error("Unhandled promise rejection:", reason);
  // log it, alert yourself, then exit cleanly in production
});

⑥ Environment variables & configuration Intermediate

Your code needs settings that change between your laptop and the real server: which port to run on, the database password, secret API keys. The golden rule of professional back-end work is: configuration and secrets do not belong in your source code. They live outside it, in environment variables.

Think of environment variables like the dials on the outside of a machine. The machine (your code) is the same everywhere, but you turn the dials differently for "test mode" vs "production mode" — without ever opening the machine up and rewiring it.

Node exposes every environment variable on the process.env object as a string. You read them like any other property:

config.js
// process.env values are ALWAYS strings (or undefined)
const PORT = process.env.PORT || 4000;          // fall back to a default
const dbPass = process.env.DB_PASSWORD;         // never hard-code this!

console.log(`Starting on port ${PORT}`);

The .env file + the dotenv package

Typing variables on the command line every time is tedious. The standard solution is a .env file holding KEY=value lines, loaded by the tiny dotenv package (npm install dotenv). It reads the file and copies each entry into process.env at startup.

.env
PORT=5000
DB_PASSWORD=sup3r-s3cret
JWT_SECRET=keep-me-private
server.js
import "dotenv/config";   // load .env BEFORE anything reads env vars

console.log(process.env.PORT);        // "5000"
console.log(process.env.DB_PASSWORD); // "sup3r-s3cret"
🔐

Why this matters so much: if you hard-code a password or API key in server.js and push it to GitHub, it's now public forever (even if you delete it later — Git keeps history). Leaked keys get scraped by bots within minutes. Always: (1) put secrets in .env, (2) add .env to .gitignore, and (3) commit a safe .env.example with blank values so teammates know which keys to set.

🔑

Load dotenv at the very top of your entry file, before any module that reads process.env. Load it late and those modules will see undefined — a frustrating, silent bug.

⑦ A web server from scratch (just to see it)

Node can be a web server with zero extra packages. You won't write servers this way in practice — but seeing it once demystifies what Express does for you.

bare-server.js
import { createServer } from "http";

const server = createServer((request, response) => {
  response.writeHead(200, { "Content-Type": "text/plain" });
  response.end("Hello from a raw Node server!");
});

server.listen(4000, () => {
  console.log("Server running at http://localhost:4000");
});

Run it with node bare-server.js, then open http://localhost:4000 in your browser. You just served a web page from your own computer! 🎉

😅

Notice there's no easy way to handle different URLs or methods here — you'd write a mountain of if statements. That pain is exactly why Express exists.

⑧ Express — the friendly web framework

Express is the most popular Node framework. It turns the messy raw server into clean, readable routes: "when a request comes to this URL with this method, run this function."

server.js — your first real server
import express from "express";

const app = express();
app.use(express.json());     // lets us read JSON request bodies

// A route: method + path + handler
app.get("/", (req, res) => {
  res.send("Welcome to my API!");
});

app.get("/api/hello", (req, res) => {
  res.json({ message: "Hello, world!", time: new Date() });
});

// :name is a URL parameter
app.get("/api/greet/:name", (req, res) => {
  res.json({ greeting: `Hi, ${req.params.name}!` });
});

app.listen(4000, () => {
  console.log("API running at http://localhost:4000");
});

Run node server.js and visit localhost:4000/api/greet/Sam. You'll get back JSON: {"greeting":"Hi, Sam!"}.

resThe response — how you reply (res.json(...), res.send(...))

These routes are your REST API. Module 4 zooms in on doing them properly — methods, status codes, and full CRUD. You've just built the foundation.

⑨ Express, deeper: reading requests & shaping responses Advanced

Once you've got routes responding, the next skill is fluently reading what the client sent and replying in exactly the right way. Every request handler gets two objects — req (what came in) and res (how you reply). Here's where the useful data lives on req:

reading-requests.js
// GET /api/tasks/42?fields=title
app.get("/api/tasks/:id", (req, res) => {
  console.log(req.params.id);     // "42"  (from the path)
  console.log(req.query.fields);  // "title" (from ?fields=...)
  res.json({ id: req.params.id });
});

// POST /api/tasks   body: { "title": "Buy milk" }
app.post("/api/tasks", (req, res) => {
  const newTask = req.body;       // { title: "Buy milk" }
  res.status(201).json(newTask);  // 201 = "Created"
});

Sending the right response

The res object has a small, friendly toolkit. The most important habit is setting a correct status code — a three-digit number telling the client what happened (you'll go deep on these next module).

res.json(obj)

Send a JavaScript object back as JSON. The bread-and-butter of an API.

res.status(code)

Set the status, then chain a body: res.status(404).json(...). Default is 200.

res.send(text)

Send plain text or HTML. Express guesses the content type for you.

res.sendStatus(204)

Set status and send a short matching body in one call (e.g. 204 No Content).

Modular routes with express.Router()

Cramming every route into server.js gets unwieldy fast. A Router is a mini-app: a self-contained bundle of related routes you define in its own file and then "mount" under a path prefix. All your task routes go in routes/tasks.js, mounted at /api/tasks — so inside that file you only write the part after the prefix.

routes/tasks.js
import { Router } from "express";
const router = Router();

router.get("/", (req, res) => {       // → GET /api/tasks
  res.json([{ id: 1, title: "Learn Node" }]);
});
router.get("/:id", (req, res) => {    // → GET /api/tasks/:id
  res.json({ id: req.params.id });
});

export default router;
server.js
import express from "express";
import taskRoutes from "./routes/tasks.js";

const app = express();
app.use(express.json());

// mount the router under a prefix:
app.use("/api/tasks", taskRoutes);

app.listen(4000);

Serving static files

Need to serve a built React app, images, or a plain HTML page? One line. express.static turns a folder into web-accessible files.

static.js
// files in ./public are served at the site root:
//   public/logo.png  →  http://localhost:4000/logo.png
app.use(express.static("public"));

404s & centralized error handling

Two pieces every real Express app needs, and they go last, after all your routes. A catch-all 404 handler runs when no route matched, and a special 4-argument error handler catches anything that threw. Express recognises the error handler purely by its four parameters — (err, req, res, next) — so the signature matters.

errors.js — place these AFTER all routes
// 1) Nothing matched → 404
app.use((req, res) => {
  res.status(404).json({ error: "Not found" });
});

// 2) Something threw → central error handler (note the 4 args!)
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ error: "Internal server error" });
});
⚠️

Order is everything in Express — it checks middleware and routes top to bottom. Put the 404 handler after your real routes (or it'll swallow every request), and the error handler dead last of all.

⑩ Middleware — code that runs in the middle Advanced

Middleware is a function that runs between the request arriving and your route handler — for logging, checking logins, parsing data, etc. express.json() above is middleware. The mental model: every request flows through a pipeline of middleware functions, each of which can inspect it, modify it, end it early, or pass it along by calling next().

You can write your own:

middleware.js
// A logger that runs on every request
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next();   // pass control to the next handler — don't forget this!
});

// A simple "auth" gate for one route
function requireKey(req, res, next) {
  if (req.headers["x-api-key"] === "secret123") {
    next();                       // allowed — continue
  } else {
    res.status(401).json({ error: "Unauthorized" });
  }
}

app.get("/api/admin", requireKey, (req, res) => {
  res.json({ data: "Top secret 🤫" });
});
⚠️

Always call next() (or send a response) in middleware. Forget it and the request hangs forever — a classic beginner bug. The browser just spins; nothing crashes, which makes it sneaky to debug.

App-level vs router-level

Where you attach middleware decides what it applies to:

  • app.use(fn) — app-level: runs for every request to the whole server (logging, CORS, body parsing).
  • router.use(fn) or app.use("/api/admin", fn) — router/path-level: runs only for requests matching that prefix (e.g. an auth gate just for admin routes).
  • app.get("/x", fn1, fn2, handler) — route-level: middleware for one specific route, listed before the final handler.

Third-party middleware you'll meet constantly

You rarely write everything yourself — the ecosystem has battle-tested middleware you just install and use:

common-middleware.js
import express from "express";
import cors from "cors";       // npm install cors
import morgan from "morgan";   // npm install morgan

const app = express();

app.use(cors());              // allow your React app to call this API
app.use(morgan("dev"));       // log every request: GET /api/tasks 200 4ms
app.use(express.json());      // parse JSON bodies into req.body

// ...your routes go here, AFTER the middleware above
🌍

What is CORS? Browsers block a page on localhost:5173 (your React app) from calling an API on localhost:4000 by default — a security rule called the same-origin policy. The cors middleware tells the browser "it's fine, I allow this," which you'll need the moment your front-end and back-end run on different ports.

⑪ A sensible project structure: separation of concerns

As your server grows, a single giant server.js becomes a nightmare to navigate. The professional cure is separation of concerns: each file has one job. Routes decide which URL does what; controllers decide what to do; services hold the business logic; the db layer talks to the database. A request flows cleanly through these layers.

Here's a clean starting layout you'll expand in the capstone:

folder structure
my-server/
├── package.json
├── package-lock.json    // exact dependency versions (commit this)
├── .env                 // secrets — NEVER commit this
├── .env.example         // blank template — safe to commit
├── .gitignore           // lists node_modules and .env
├── server.js            // entry point: wires it all together
├── routes/
│   └── tasks.js         // URL → controller mapping
├── controllers/
│   └── taskController.js // request/response handling
├── services/
│   └── taskService.js   // business logic (no req/res here)
├── db/
│   └── pool.js          // database connection
└── node_modules/        // installed packages (auto-generated)
🧩

The big win of this layout: each file is small and testable, and you can reason about one concern at a time. A new teammate can guess exactly where a piece of code lives. Don't over-engineer a tiny script with all five layers — but as soon as a file passes ~150 lines, start splitting.

⑫ Auto-reload & debugging your Node app

Stop restarting by hand: auto-reload

Restarting the server after every edit gets old in about five minutes. Two ways to make it automatic:

Built-in --watch

Modern Node (18+) has it built in: node --watch server.js. No install needed. Restarts on save.

nodemon

The long-time favourite: npm install -D nodemon, then nodemon server.js. More configurable; works on older Node too.

Wire whichever you choose into a dev script so you just type npm run dev:

package.json scripts
"scripts": {
  "start": "node server.js",          // production: run once
  "dev": "node --watch server.js"     // development: auto-restart
}

Reading a stack trace

When Node crashes, don't panic at the wall of red text — it's a stack trace, and it's trying to help. Read it top to bottom: the first line is what went wrong, and the first at … line pointing at your file (not deep inside node_modules) is usually where.

a typical stack trace
TypeError: Cannot read properties of undefined (reading 'name')
    at getUser (/app/controllers/userController.js:12:18)  ← your bug: file:line:col
    at /app/routes/users.js:7:5
    at Layer.handle (/app/node_modules/express/lib/router/layer.js:95:5)
  • Line 1 names the error type and message — here, something was undefined when you tried to read .name.
  • The first at with your path is the scene of the crime: userController.js, line 12, column 18.
  • Lines pointing into node_modules are usually just the framework's plumbing — skip past them.

Two debugging tools

  • console.log — humble but mighty. Sprinkle logs to see what a variable actually holds. console.log({ req: req.body }) labels the output so you don't lose track.
  • node --inspect server.js — opens a real debugger you can attach to from Chrome DevTools or VS Code, letting you set breakpoints and step through line by line. Reach for it when console.log isn't enough.
💡

VS Code can launch Node with the debugger attached at the press of F5 — breakpoints, variable inspection, the works. Once you've debugged a tricky bug this way once, you'll never want to go back.

⑬ What you've built is (almost) a REST API

Take a step back — those app.get and app.post routes returning JSON? That's the skeleton of a REST API: a standard, predictable way for clients (your React app, a mobile app, anyone) to read and change data over HTTP.

Key takeaways

  • Node.js is a runtime (built on Chrome's V8 engine) that runs JavaScript on the server — same language, plus access to files, network and databases; no window or document.
  • The event loop makes Node non-blocking: it handles many connections on one thread by doing slow I/O in the background — so never block it with heavy synchronous work.
  • Modules split code across files. Prefer modern ES Modules (import/export, needs "type":"module") over older CommonJS (require).
  • Node ships with core modules — fs, path, os, url, http, events, process — before you install anything.
  • npm installs packages; package.json lists dependencies vs devDependencies, semver (^ ~) controls upgrades, and package-lock.json pins exact versions. Never commit node_modules.
  • Back-end work is async — callbacks → promises → async/await. Wrap awaits in try/catch and handle rejections.
  • Keep config & secrets in environment variables (.env + dotenv, git-ignored), read via process.env.
  • Express turns requests into clean routes with req (.params, .query, .body) and res (.json, .status). Use express.Router(), express.static, a 404 handler, and a 4-arg error handler.
  • Middleware runs in a pipeline between request and handler — order matters, always call next(). Lean on cors, morgan, and express.json().
  • Organise real apps by separation of concerns (routes / controllers / services / db), auto-reload with --watch or nodemon, and read stack traces top-down to debug.
