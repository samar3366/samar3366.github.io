Tool Use & Function Calling

An LLM is a brain in a jar. Tools give it hands. The standard mechanism — function calling — is simple, elegant, and the basis for nearly every agentic system today.

Kid Translation: Giving the AI a Toolbox

🧒 Kid Translation

An LLM by itself can only do one thing: produce text. It can't check the weather, look up a stock price, run a calculation, or send an email. It just talks.

So we give it a toolbox. We say: "Hey, here are 5 tools you can use. Tool 1 is get_weather(city). Tool 2 is send_email(to, subject, body). When you want to use one, write a special message that says which tool and what arguments. I'll run it for real and tell you what happened. Then you can keep going."

The model learned during training how to use tools this way. So now when you ask "what's the weather in Tokyo?", it doesn't make up an answer — it generates a tool call. We run the actual API. We feed the result back. The model writes a real, current answer based on real data.

That's function calling. Brain stays in the model. Hands are the tools. The protocol between them is just structured text.

The Tool Schema

Each tool you give the model is described by a schema: name, description, and the parameters it accepts. The standard format is JSON Schema. Here's a simple example:

— Tool Schema (JSON) —
{
  "name": "get_weather",
  "description": "Get current weather for a city",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {"type": "string", "description": "City name"},
      "units": {"type": "string", "enum": ["c", "f"]}
    },
    "required": ["city"]
  }
}

The schema isn't just for documentation — it's also constraint. Modern model APIs validate tool calls against the schema and reject malformed ones. For OpenAI's structured outputs and Anthropic's tool use, the model is biased at decode time to produce only schema-conforming JSON. You get type safety for free.

The description fields are the most underrated part. The model uses them to decide which tool to call. A vague description ("Get info") leads to wrong tool selection. A precise one ("Returns current temperature, conditions, humidity for any city worldwide; do not use for forecasts") prevents misuse.

The Tool-Use Loop

The loop has four steps. The host code (your application) runs the loop; the model only ever generates text.

For multi-step tasks the loop iterates: the model can call multiple tools in a row, with each result informing the next call. Each iteration is one LLM call. This is exactly the agent loop from Chapter 19, made concrete.

Parallel & Chained Tool Calls

Modern model APIs (OpenAI, Anthropic, Gemini) support parallel tool calls: in a single response, the model can request multiple tool calls at once. The host runs them all (often concurrently), and feeds all results back together.

This matters for performance. If a query needs the weather in three cities, sequential calls = three round trips = ~3 seconds. Parallel calls = one round trip = ~1 second. For agents that frequently fan out (research, comparison, multi-source aggregation), this is a major speedup.

Chained calls are when the result of one tool becomes the argument of the next. Models handle this naturally — call search_users("alice"), get back a user_id, then call get_orders(user_id). The model holds the chain in its working memory and doesn't need explicit guidance.

Error Handling & Retries

Tools fail. Networks time out, APIs return 500s, schemas mismatch. Production tool-use systems need explicit error handling:

  • Return errors as data, not exceptions. When a tool fails, return a structured error message to the model: {"error": "rate_limit", "retry_after_seconds": 30}. The model can read it and decide what to do.
  • Let the model retry. Often the right behavior — but with limits. After 3 failures, escalate or give up.
  • Distinguish recoverable from fatal. A 401 ("unauthorized") usually means stop. A 503 ("temporarily unavailable") means retry.
  • Sanitize inputs. Even if the model produces schema-valid JSON, it might pass nonsensical values. Validate semantically before executing.
  • Log every call. For debugging, audit, and post-mortem when something goes wrong.

Designing Good Tools

★ Principles for tool design

Few, atomic, well-named. Five clear tools beat fifty muddled ones. Models pick wrong tools when the boundaries blur.

Strong descriptions. The description tells the model when to use the tool. Spend more words here than you think you need.

Constrained parameters. Use enums and required fields. Don't accept "any string" if the value should be one of three options.

Idempotent when possible. If the model accidentally calls send_email twice, did the user get two emails? Make it safe to retry.

Return structured data. Models work better with structured JSON results than long prose.

Bound the output. A tool that returns 100,000 tokens of database data will blow up your context. Truncate or summarize.

Tool use is what made LLMs go from "write me a paragraph" to "do my taxes." The protocol is simple JSON Schema, but the design discipline behind a good tool set is where agent reliability lives or dies.
💼 Pro Talk · How to sound fluent

"Function calling" (OpenAI's term) and "tool use" (Anthropic's term) are the same thing. "Tool calling" is increasingly the neutral term.

"JSON Schema" — the standard format for tool parameters. "Structured outputs" — the broader feature of constraining LLM output to a JSON schema, of which tool calling is a use case.

"Strict mode" — when the model is grammar-constrained at decode time to only emit schema-valid JSON. Reliable, but limits some flexibility.

"Parallel function calls" — the model emits multiple tool calls in one response, executed concurrently.

"Toolformer" — early research on teaching models to use tools (2023 Meta paper). Mostly historical now; tool use is fully native to frontier models.

"Tool poisoning" — a security risk where an attacker manipulates tool descriptions to get the model to misuse tools.

Self-Check Questions

🧠 Test yourself · Answer out loud
  1. What does the LLM actually do when it "calls" a tool?
    Answer

    It generates a structured message (JSON) saying which tool and what arguments. The host application reads that message and actually executes the tool. The LLM never directly executes anything.

  2. Why are tool descriptions important?
    Answer

    Because the model uses them to decide which tool to call. Vague descriptions lead to wrong selections. A precise description includes when to use the tool, what it returns, and any caveats.

  3. How many LLM calls happen for one tool-using request?
    Answer

    At least two: one where the model decides on the tool (and emits the call), one where it composes a final answer from the tool's result. Multi-step tasks have more.

  4. What's the benefit of parallel tool calls?
    Answer

    Latency. Multiple independent tool invocations run concurrently in one round trip instead of sequentially. Critical for agents that fan out across many sources.

  5. Why is idempotency a desirable property of write-action tools?
    Answer

    Because models occasionally retry, double-call, or re-invoke tools — and a write that's accidentally called twice should not double-charge the user, send two emails, etc. Idempotent design makes retries safe.
