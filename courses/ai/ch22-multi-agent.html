Multi-Agent Systems

Several agents collaborating like a team. Sometimes spectacular, sometimes catastrophic. Knowing when multiple agents help — and when one is enough — is one of the most useful judgments in modern AI design.

Kid Translation: A Crew, Not a Soloist

🧒 Kid Translation

Imagine you have to make a movie. One person could do everything — write the script, act, film, edit, do the music. Possible, but slow and probably not great.

Or you can have a crew: a director who coordinates, a writer who writes, an actor who acts, an editor who edits. Each person specializes. The director hands off jobs and combines results. The movie comes out faster and better.

Multi-agent AI works the same way. Instead of one big agent trying to do everything, you have several smaller, specialized agents. One researches. One writes. One reviews. One coordinates them all. Each does its job well; the coordinator stitches the work together.

The catch: crews can also bicker, get confused, redo each other's work, or get lost in committee. Adding more agents doesn't always help. The trick is knowing when a crew beats a soloist.

Why Multi-Agent?

There are four legitimate reasons to use multiple agents instead of one big one:

  1. Specialization. A "researcher agent" with web search tools does different work than a "writer agent" with style guides. Specialized prompts, tools, and even different models per role can outperform a single generalist agent.
  2. Parallelism. Many tasks have independent sub-tasks: research five competitors, summarize ten documents, audit twenty pull requests. Spawning agents that run in parallel can finish in 1/N the wall time.
  3. Context isolation. Each agent has its own context window. A researcher's 100K-token research isn't dumped into the writer's working memory — only the summary they need is. Avoids "lost in the middle" and reduces cost.
  4. Critique & verification. One agent generates, a separate agent critiques. The critic isn't biased by the generator's reasoning. Patterns like reviewer-loops, debate, and self-consistency exploit this.

Core Patterns

The multi-agent landscape boils down to a small number of recurring patterns:

PatternWhat it looks likeUse when
Orchestrator–WorkerOne coordinator agent decomposes a task and assigns chunks to specialized workers.Most production cases. Clear hierarchy, accountable.
PipelineAgents run in a fixed sequence. Output of one is input to the next.The task naturally splits into phases (research → outline → draft → review).
Parallel fan-outOne agent spawns N parallel workers, then aggregates their outputs.Independent sub-tasks (compare 10 vendors, summarize 5 papers).
Critic / reviewer loopA generator agent and a critic agent alternate until quality threshold.Code review, writing, anywhere quality matters more than speed.
DebateMultiple agents argue different positions; a judge picks or synthesizes.Research, decision-making, reducing single-agent bias.
Swarm / societyMany peer agents communicate without strict hierarchy.Mostly research. Hard to control in production.

Orchestrator–Worker: The Workhorse

The orchestrator–worker pattern dominates production multi-agent systems. It's worth understanding well.

Anthropic published a detailed write-up of how their Research feature uses orchestrator–worker. The orchestrator plans the research direction and dispatches sub-agents to investigate parallel angles. Each sub-agent has its own context, runs many tool calls in parallel, and reports a tight summary back. The orchestrator stitches summaries into the final report. They reported ~3× better results than a single-agent approach on complex research tasks — at substantially higher token cost.

Real production multi-agent systems in 2026: research assistants (OpenAI Deep Research, Perplexity Deep Research, Anthropic Research), coding agents (Devin, Claude Code, Cursor's background agents), customer-service flows that escalate between specialized agents, and behind-the-scenes orchestration in many enterprise AI deployments.

When NOT to Use Multi-Agent

Multi-agent is overused. The default for most tasks should be a single agent. Add agents only when you have a specific reason. Some warning signs that you're over-architecting:

  • The task fits in one context window comfortably. Just use one agent.
  • The sub-tasks aren't independent. If every "worker" needs to know what every other worker did, the coordination cost dominates.
  • You're hoping multiple agents will catch each other's errors. Sometimes works, but fragile — agents share the same biases of the underlying model.
  • You can't articulate why a single agent wouldn't do this. If you can't explain the specific benefit (specialization, parallelism, context isolation, verification) — there isn't one.
⚠ The 10× cost surprise

Multi-agent systems can easily cost 5–15× more in tokens than equivalent single-agent solutions, because every sub-agent re-reads context, every handoff repeats setup, and orchestrators re-process intermediate results. If you go multi-agent, measure cost per task from day one.

⚠ Communication degradation

Every handoff between agents loses information. Sub-agent A finds 50 facts; the summary it sends to the orchestrator captures 10. Sub-agent B then works from that compressed context. Multi-agent systems can converge on superficially-correct, deep-down-impoverished answers if you don't watch for this.

Multi-agent is a power tool, not a default. Start with one strong agent and rich tools. Add agents only when you can name the specific capability the second agent is bringing — specialization, parallelism, isolation, or independent critique.
💼 Pro Talk · How to sound fluent

"Orchestrator" / "controller" / "conductor" — the coordinator agent. "Worker" / "sub-agent" — the delegate.

"Handoff" — passing control from one agent to another. "Context isolation" — the practice of not sharing full context between agents.

"Agentic workflow" — sometimes used loosely; technically means a process where one or more agents drive multi-step execution, possibly with human checkpoints.

"Reflexion" — the self-critique pattern (single-agent variant of critic loops).

"Multi-agent debate" — research pattern where agents argue different sides; reduces hallucination on factual questions.

"Swarm intelligence" — broader term, often more aspirational than practical in production.

"Agent2Agent (A2A) protocol" — Google's proposed standard for agent-to-agent communication, complementary to MCP. Early-stage as of 2026.

Self-Check Questions

🧠 Test yourself · Answer out loud
  1. Name the four legitimate reasons to go multi-agent.
    Answer

    Specialization, parallelism, context isolation, and independent critique/verification.

  2. What's the orchestrator–worker pattern?
    Answer

    One coordinator agent owns the goal and decomposes it into sub-tasks. Specialized worker agents handle each sub-task independently. The orchestrator integrates their results into the final answer.

  3. Why might multi-agent systems cost 10× more than single-agent ones?
    Answer

    Every sub-agent re-reads context, handoffs duplicate setup, and orchestrators re-process intermediate results. Token costs compound across roles even when the same model is used throughout.

  4. What's "communication degradation" in multi-agent systems?
    Answer

    Information loss at each handoff. A sub-agent's full context gets compressed into a summary for the next agent. Across many handoffs, important nuance can be lost while the system still produces plausible-looking outputs.

  5. When should you NOT reach for multi-agent?
    Answer

    When a single agent with rich tools could do the task within one context window, when sub-tasks aren't independent, or when you can't articulate a specific reason a second agent is helping. Multi-agent is a power tool, not a default.
