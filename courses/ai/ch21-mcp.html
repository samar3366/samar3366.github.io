MCP: Model Context Protocol

If function calling lets one app give one model some tools, MCP makes those tools portable. It's the USB-C of AI integrations: write the connector once, plug it into any compatible client.

Kid Translation: One Plug Fits All

🧒 Kid Translation

Remember when every electronic device used a different charger? Phones had one plug, cameras had another, laptops had another. Then USB-C came along and suddenly every device used the same plug.

Before MCP, every AI app had its own way of plugging tools in. ChatGPT had plugins. Claude had its own integrations. Cursor had another. Gmail tools had to be rewritten for each one. If you wrote one tool, you could only use it in one place.

MCP is USB-C for AI tools. Build your "Connect to Slack" tool once, in the MCP shape. Now Claude, Cursor, your custom app, anything that "speaks MCP" can plug into your Slack tool without modification. Same plug, every device.

Anthropic launched it in late 2024. By 2026 it had become a de facto industry standard — OpenAI, Google, every major framework adopted it. The boring infrastructure that quietly makes the agent ecosystem possible.

The M×N Integration Problem

Imagine M AI applications (Claude, ChatGPT, Cursor, custom internal tools — pick a number) and N data sources or services (Gmail, Slack, GitHub, your CRM, your database — pick a much bigger number). Without a standard, you need to write M × N integrations: one for each pair. That's quadratic. It doesn't scale.

With a standard protocol, you write M clients (each AI app implements MCP once) and N servers (each tool exposes itself via MCP once). The total integration work drops from M × N to M + N — linear. Anyone who builds either side gets compatibility with everyone on the other side, automatically.

What MCP Is

MCP is a protocol specification. Not a framework, not a library, not a service. It's a written agreement on how AI applications can talk to external tools and data sources. The spec covers:

  • Message format. Based on JSON-RPC 2.0 — a simple, well-known standard.
  • Transports. stdio (for local processes), HTTP/SSE (for remote services), and Streamable HTTP (for newer remote use).
  • Capabilities. What kinds of things can be exposed (tools, resources, prompts — covered below).
  • Lifecycle. How clients and servers handshake, list capabilities, and shut down.

Anthropic released MCP open-source in November 2024, with an open governance model. It's now stewarded by an open-source community, with multiple SDKs (Python, TypeScript, Go, Rust, others) and a growing ecosystem.

Architecture: Clients & Servers

MCP has two roles:

  • MCP server. Wraps a specific tool, data source, or capability. A "GitHub MCP server" exposes GitHub APIs. A "filesystem MCP server" exposes file read/write. Each server is small and focused.
  • MCP client. The AI application — Claude Desktop, Cursor, your custom agent — that connects to one or more servers and uses their capabilities.

An AI app typically connects to several servers at once. Claude Desktop on a developer's laptop might be wired up to: a filesystem server (read/write project files), a GitHub server (PRs, issues), a Postgres server (run queries against a dev DB), a Slack server (post messages). The user just adds servers to a config file; the app discovers their capabilities automatically.

The Three Primitives

An MCP server can expose three kinds of capabilities. This is the core of the spec.

PrimitiveWhat it isExample
ToolsFunctions the model can call to do things or fetch data — exactly like Chapter 20 tool use.create_github_issue(repo, title, body)
ResourcesRead-only data the model can be given as context. Files, DB rows, web pages — addressed by URI.file:///project/README.md
PromptsReusable prompt templates the user (not the model) can invoke. Like saved snippets or slash-commands./summarize-pr with a configured prompt

Tools are the most common — they're how the model takes actions. Resources are useful for "here are some files for context" workflows. Prompts are the least-used but elegant: a server can ship best-practice prompts alongside the tools they go with.

Two-way communication: servers can also request things from clients ("ask the user for input," "ask the model to summarize this"). This makes possible patterns like servers that drive the agent loop themselves.

Real MCP Servers

By 2026 there are thousands of MCP servers, official and community-built. A non-exhaustive sampling:

  • Official reference servers (Anthropic-published) — filesystem, git, GitHub, GitLab, Slack, Google Drive, PostgreSQL, SQLite, Sentry, Puppeteer (browser automation).
  • Vendor-published — Stripe, Linear, Notion, Atlassian (Jira/Confluence), Cloudflare, AWS, Figma, Sentry, PayPal, and most major SaaS vendors now ship official MCP servers.
  • Community servers — anything you can think of: Spotify, Apple Notes, Home Assistant, Kubernetes, Terraform, HackerNews, weather APIs, finance data, mathematical packages, and on and on.
  • Internal company servers — wrap your internal services so your AI tools can read your wiki, query your data warehouse, deploy to your infra. The biggest enterprise use case.

Discoverability is improving fast. Registries like mcp.so and Anthropic's official directory let you browse available servers and add them with one click in supported clients.

MCP vs. Plain Function Calling

AspectFunction calling (Ch 20)MCP (this chapter)
ScopePer-application. You define tools in your code.Cross-application. Tools are exposed by separate servers.
PortabilityNone. Your tools live inside your app.High. Any MCP client can use any MCP server.
SetupLower friction for small apps.More moving parts, but pays off as you grow.
MechanismDirect function call inside your process.JSON-RPC over stdio or HTTP between processes.
Best forTightly coupled, custom in-app tools.Reusable integrations; corporate stacks; user-extensible apps.

They're complementary, not competing. Under the hood, when an MCP client calls an MCP server's tool, the model still uses Chapter 20-style function calling — MCP just standardizes how the tool definition gets there. Most modern AI applications use both: native function calling for app-specific stuff, MCP for everything that should be portable.

MCP is boring infrastructure that does enormous work. Just like USB-C didn't change what your devices do, but radically changed who can use them with what — MCP doesn't change what AI can do, it changes who can connect what to what.
💼 Pro Talk · How to sound fluent

"MCP server" — exposes capabilities. "MCP client" — uses capabilities. "MCP host" — the parent app the client lives inside (e.g., Claude Desktop is a host that contains the MCP client).

"Tools / resources / prompts" — the three primitives.

"Stdio transport" — for local servers; the host spawns the server as a child process and pipes JSON-RPC over stdin/stdout. "Streamable HTTP" — for remote servers.

"Capability negotiation" — the handshake where client and server agree on what features each supports.

"Sampling" (in MCP) — when a server asks the client to perform an LLM call on its behalf. Lets servers leverage the host's model without bringing their own keys.

"Roots" — paths or URIs the client tells the server it's allowed to access (security boundary).

Self-Check Questions

🧠 Test yourself · Answer out loud
  1. What problem was MCP designed to solve?
    Answer

    The M×N integration problem: every AI app had to build its own integrations to every tool. MCP standardizes the interface so each side only implements one protocol — turning quadratic integration work into linear.

  2. What are the three primitives an MCP server can expose?
    Answer

    Tools (callable functions), resources (read-only context data, addressed by URI), and prompts (reusable prompt templates the user can invoke).

  3. What's the difference between an MCP server and an MCP client?
    Answer

    The server provides capabilities (wraps a tool, data source, etc.). The client is the AI application that connects to one or more servers to use their capabilities. The host is the broader app the client runs inside.

  4. Does MCP replace function calling?
    Answer

    No, it builds on it. The model still uses native function calling to invoke MCP tools. MCP just standardizes how tools are defined, hosted, and discovered, so the same tools work across many AI clients.

  5. Why does MCP matter to enterprises specifically?
    Answer

    Enterprises have many internal services they want AI tools to access. Building one MCP server per service exposes them to all current and future AI applications — Claude, custom agents, IDEs, future products — without per-app integration work.
