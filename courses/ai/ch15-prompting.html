Kid Translation: How You Ask Matters

🧒 Kid Translation

You know how if you say "Mom, can I have a cookie?" you might get one — but if you say "Mom, can I PLEASE have just one cookie before dinner because I had a really good day at school and didn't even argue with my brother?" — you definitely get the cookie?

It's the same with AI. The same question can get you a lazy, generic answer or a fantastic, detailed one — depending on how you ask. The trick isn't being polite (though that doesn't hurt). It's being clear, specific, and giving examples of what you want.

Prompt engineering is just learning to ask well. There's no magic spell — the techniques are practical and learnable. Once you know them, you'll get noticeably better results from every AI you use.

Anatomy of a Good Prompt

A high-quality prompt typically has these parts. Not all are needed every time, but knowing them gives you a checklist.

  1. Role / context. Tell the model who it's playing or what context to assume. "You are a senior security engineer reviewing code for vulnerabilities."
  2. Task. What you want, specifically. Not "help with my code" — "find any SQL injection risks in this function and explain each one."
  3. Inputs. The actual content the model needs to work with — code, document, data — clearly delimited.
  4. Constraints. Length, tone, format, what to avoid. "Use bullet points. Maximum 5. No code in the response."
  5. Examples (optional but powerful). Show 1–3 example inputs and outputs in the format you want.
  6. Output format. JSON? Markdown? Structured headings? Specify.
Bad → Good prompt comparison

Bad: "review my code"

Good: "You are a senior Python engineer doing a code review. Below is a function for processing user uploads. Identify (1) any security risks, (2) edge cases that aren't handled, (3) anything that would fail at scale. Format your response as three sections with bullet points. Maximum 8 bullets total. <code>...</code>"

The good version gives the model role, task, structure, and constraints. The output will be 5× more useful.

Core Patterns: Zero-Shot, Few-Shot, Chain-of-Thought

Three foundational patterns every practitioner should know:

Zero-shot: just ask

Just describe the task and let the model do it from its general training. Works for tasks that the model has seen many examples of during training. Best for common tasks.

Few-shot: show examples

Include 1–5 example input/output pairs in the prompt. The model learns the pattern from your examples and applies it to the new input. Powerful for tasks with specific formats, edge cases, or domain conventions.

— Few-Shot Pattern —
Input: "I love this!" → Sentiment: positive
Input: "Worst purchase ever." → Sentiment: negative
Input: "It was okay." → Sentiment: neutral
Input: "Couldn't be happier." → Sentiment: ?

Chain-of-thought: ask for reasoning

For complex problems, ask the model to think through the steps before answering. We covered this in Chapter 11 — for non-reasoning models, simply adding "Let's think step by step" or "First, list the key facts. Then reason through them. Then answer." dramatically improves accuracy.

For modern reasoning models (Claude with extended thinking, OpenAI o-series), this often happens automatically — but explicit guidance still helps shape the reasoning.

System Prompts & Roles

Most chat APIs have two kinds of messages: system prompts (set up for the conversation) and user messages (what the user asks).

System prompts are powerful because they carry persistent context that shapes the entire conversation:

  • Identity / persona ("You are a careful financial advisor named Iris.").
  • Capabilities and limits ("You may not give specific stock picks. You always recommend consulting a licensed advisor.").
  • Default behaviors ("Always cite sources. Always explain your reasoning before conclusions.").
  • Format preferences ("All responses in markdown. Code in fenced blocks.").

For developer/API use, the system prompt is your main lever. For consumer chat (ChatGPT, Claude.ai), users typically can't set system prompts directly — but features like Custom Instructions or Projects effectively serve the same role.

Structured Outputs

For programmatic use, you usually want the model to return data in a predictable format — JSON, XML, a specific schema. Two ways:

  1. Tell the model the schema. "Return your answer as JSON with these exact keys: title (string), summary (string, max 100 words), tags (array of strings)." Modern models follow schemas reliably.
  2. Use API-level structured output features. OpenAI's "structured outputs" mode, Anthropic's tool use, and many model providers now offer guaranteed JSON-conforming output by constraining the decoder. When available, use it.
★ Pro tip

For complex schemas, give a complete example output. Models follow examples more reliably than schema specs. "Here's an example of correctly formatted output: {...}" works better than abstract type descriptions.

Model-Specific Quirks

Different models have different prompting cultures. Some patterns:

  • Claude responds well to XML tags as delimiters (<document>, <example>) and to natural conversational instructions. Prefers a clear role/task structure.
  • GPT models are happy with markdown delimiters and explicit numbered instructions.
  • Reasoning models (o-series, Claude extended thinking) need less hand-holding on reasoning steps but still benefit from clear task statements.
  • Open models (Llama, Mistral, Qwen) often need more explicit examples and stricter formats; their instruction following varies more by size and fine-tune.

The bigger principle: iterate. Run your prompt, look at the output, identify what's missing or wrong, refine. Three iterations on a prompt almost always beats one shot.

Prompt engineering is mostly clarity. The best prompts are clear about role, task, inputs, constraints, and format. The fancy techniques (CoT, few-shot, structured outputs) just add precision.
💼 Pro Talk · How to sound fluent

"System prompt" — the persistent setup. "User prompt" — what the end-user types.

"Few-shot" — including N example pairs. "Zero-shot" — no examples. "One-shot" — one example.

"Prompt template" — a reusable prompt with variable slots, used in production code.

"Prompt injection" — when an untrusted input (like a webpage being summarized) contains instructions that hijack the model's behavior. Major security topic — see Chapter 23.

"Jailbreak" — prompts designed to bypass a model's safety training. Active arms race between attackers and defenders.

"Meta-prompting" — using AI to write or improve prompts for other AI calls.

"Prompt chaining" — breaking a complex task into multiple sequential prompts where one's output feeds another's input.

Self-Check Questions

🧠 Test yourself · Answer out loud
  1. What are the typical components of a well-structured prompt?
    Answer

    Role/context, task, inputs (clearly delimited), constraints, examples (optional), and desired output format.

  2. When does few-shot prompting help most?
    Answer

    For tasks with specific formats, niche conventions, or unusual edge cases. The examples teach the model the pattern faster than describing it abstractly.

  3. Why are positive instructions usually better than negative ones?
    Answer

    Negative instructions ("don't be long") leave too much room for interpretation. Positive ones ("maximum 3 sentences") are concrete and verifiable. Models also implicitly weight positive instructions more strongly.

  4. What's a system prompt and why does it matter?
    Answer

    A persistent message that sets up role, behaviors, and constraints for the whole conversation. It shapes everything that follows and is the most powerful lever in API use.

  5. How do you make a model reliably return JSON?
    Answer

    Either: (1) use the API's structured-output mode if available, or (2) tell the model the exact schema and provide a complete example output. For complex schemas, examples beat descriptions.
